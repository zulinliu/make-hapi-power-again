## Top 3 Priority Fixes

1. **ContextMenu touch target minHeight 36px** -- Fails mobile accessibility (recommended 44px). Fix: Change `minHeight: 36` to `minHeight: 44` at ContextMenu.tsx:143.
2. **Dual h1 headings on login page** -- Both `login-brand-mobile-name` and `login-form-title` render as `<h1>` simultaneously (one is CSS-hidden, but screen readers may still encounter both on desktop). Fix: Change the mobile-brand heading to `<h2>` or use `aria-hidden` when hidden.
3. **Arbitrary pixel spacing values in Tailwind components** -- 20+ instances of `text-[10px]`, `text-[11px]`, `px-1.5`, `py-0.5`, etc. scattered across settings and tool card components. Fix: Map these to `--hp-text-xs` or create additional spacing token variants.

## Detailed Findings

### Pillar 1: Copywriting (3/4)

**Positive findings:**
- All login page strings use `t()` calls via the i18n system (LoginPrompt.tsx lines 78-151).
- English locale has 721 entries, Chinese locale has 723 entries, both with comprehensive coverage.
- Error messages are specific and actionable: `login.error.enterToken` ("Please enter an access token"), `login.error.authFailed` ("Authentication failed").
- Loading states use ellipsis convention: "Signing in...", "Loading session...".
- Empty states are descriptive: `sessions.empty.hint` ("Start a coding session in any folder under your workspace, or browse the tree first.")

**Issues found:**
- Generic "Save" and "Cancel" labels remain in ProviderSettings.tsx (lines 96, 468-471) -- these are context-ambiguous. The i18n keys `button.save` and `button.cancel` exist but "Save" for a provider form vs "Save" for a file rename are the same generic string.
- `login.help` = "Needs help?" is grammatically awkward; the conventional form is "Need help?" (en.ts:31).
- `dialog.archive.description` includes the phrase "This will disconnect active session" -- missing article "the" (en.ts:119).

### Pillar 2: Visuals (3/4)

**Positive findings:**
- Split layout (55fr/45fr grid) creates clear visual hierarchy with brand panel left, form panel right (login.css:9-10).
- Brand panel has layered visual depth: radial gradients, noise texture overlay, z-indexed content (login.css:28-55).
- Form panel uses staggered animation delays (0.15s, 0.2s, 0.5s, 0.7s) for progressive disclosure.
- SessionHeader uses semantic `<header>` and `<nav>` elements with proper `aria-label` on navigation (SessionHeader.tsx:217).
- All icon-only buttons have `aria-label` attributes (SessionHeader.tsx lines 186, 225, 237, 251, 266, 281).
- Login form has clear visual flow: label -> input -> error -> button -> links -> footer.

**Issues found:**
- ContextMenu items use `minHeight: 36` (ContextMenu.tsx:143), which is below the 44px mobile touch target standard. The login form inputs correctly use 48px desktop / 44px mobile, but the context menu regresses.
- The brand panel `aria-hidden="true"` (LoginPrompt.tsx:157) correctly hides decorative content, but the mobile brand section inside the form panel does NOT have `aria-hidden` -- it repeats the same logo and title already in the form area, creating potential screen reader duplication.
- No visual loading skeleton for the login form itself -- if fonts or CSS load slowly, the form appears unstyled. The `Spinner` component is used for submit state, but there is no initial page load skeleton.

### Pillar 3: Color (3/4)

**Positive findings:**
- Design tokens are pervasively adopted: 546 `--hp-*` occurrences across 70 component files.
- login.css references 94 `--hp-*` tokens -- the login page is fully token-driven.
- Color system uses oklch perceptual uniformity (tokens.css lines 12-55, 99-126).
- Dark mode is complete with separate token values (tokens.css:98-127).
- login.css dark mode properly overrides brand panel background with oklch values (login.css:315-323).

**Issues found:**
- FileIcon.tsx contains 16 hardcoded hex colors for file type badges (lines 4-20): `#3178c6`, `#f7df1e`, `#2563eb`, `#db2777`, etc. These are decorative but bypass the token system. While arguably acceptable for file type identification (matching language conventions), they do not respect dark mode.
- login.css hardcodes oklch values directly for radial gradients and shadows instead of using variables (lines 33-34, 66, 316, 321-322, 335, 521). These are visual-only (no token equivalent exists for gradient stops), so this is a minor deviation.
- Mermaid diagram component uses 9 hardcoded hex colors (mermaid-diagram.tsx:30-38): `#323843`, `#edf1f5`, `#6d8fd6`, etc. These are not tokenized and will not adapt to theme changes.
- CodeEditor.tsx hardcodes `#1e1e1e` background and `#888` text color (line 99) -- a monaco editor fallback that ignores the theme system.
- `--app-*` legacy variables persist in 53 files (322 occurrences) alongside `--hp-*` tokens. This dual system creates confusion about which is canonical.

### Pillar 4: Typography (3/4)

**Positive findings:**
- Geist Sans and Geist Mono font files are self-hosted in `public/fonts/` (71KB and 71KB respectively).
- Source Serif 4 is loaded from Google Fonts CDN as the display font (typography.css:29).
- Typography tokens define 7 responsive sizes via `clamp()` (typography.css:38-44).
- Font weights are tokenized: normal (400), medium (500), semibold (600), bold (700) (typography.css:54-57).
- login.css uses `font-family: var(--hp-font-display)` for headings and `var(--hp-font-sans)` for body text.

**Issues found:**
- Tailwind components use 7 distinct font size classes: `text-xs`, `text-sm`, `text-base`, `text-lg`, `text-xl`, `text-2xl`, `text-3xl`. This exceeds the recommended maximum of 4 without a UI-SPEC declaring this intentional.
- Two arbitrary font sizes bypass both Tailwind and token systems: `text-[10px]` (files.tsx:129) and `text-[11px]` (files.tsx:141, ToolGroupCard.tsx:24, login.css:394).
- Three font weights dominate: `font-medium` (most frequent by far), `font-semibold`, and `font-bold`. The `font-medium` usage is excessive -- 140+ instances. This flattens the typographic hierarchy because everything is medium weight, making it hard to distinguish label text from body text.
- login.css uses `font-size: 13px` directly for labels (line 183) and `font-size: 12px` for footer (line 309) instead of the typography tokens.

### Pillar 5: Spacing (2/4)

**Positive findings:**
- tokens.css defines a clean 12-step spacing scale from 2px to 48px (lines 66-76).
- login.css uses consistent spacing aligned to the token scale: 48px padding, 24px gaps, 12px margins, 16px padding.
- login.css responsive breakpoints are well-structured: mobile (<767px), tablet (768-1024px), desktop (>768px), narrow mobile (<400px).
- `env(safe-area-inset-*)` is used for mobile notch/home bar accommodation (login.css:509, 548, 557, 603).

**Issues found:**
- Tailwind components use 20+ arbitrary pixel values in bracket notation: `text-[10px]`, `text-[11px]`, `px-1.5`, `py-0.5`, `min-h-[260px]`, `max-h-[45vh]`, `min-h-[88px]`, `min-w-[160px]`, `min-w-[140px]`, `min-w-[220px]`, `min-w-[170px]`, `min-w-[230px]`, `w-[4rem]`, `rounded-[20px]`, `rounded-[16px]`, `border-l-[3px]`, `p-[2px]`, `rounded-[22px]`, `border-[1.5px]`. These are spread across settings, tool cards, and file management components.
- settings/index.tsx repeats nearly identical dropdown class strings 7 times (lines 439, 494, 542, 590, 645, 705, 753) with only `min-w-[...]` varying. This is a DRY violation that also introduces spacing inconsistency risk.
- login.css uses pixel values directly (48px, 32px, 24px, 20px, 16px, 12px, 10px, 8px) rather than `var(--hp-space-*)` tokens. While these map to the same values, the indirection would ensure consistency if the token scale changes.

### Pillar 6: Experience Design (3/4)

**Positive findings:**
- Loading states: Spinner component used in LoginPrompt submit button (line 229), FileContentSkeleton and FileListSkeleton for file views, LoadingState component for session/git pages.
- Error states: Login form displays errors via `role="alert"` divs (LoginPrompt.tsx:216, 284). Error states for file operations, git commands, and API calls are handled with specific i18n messages.
- Empty states: Sessions empty state component (SessionList.tsx:30), files empty states with actionable CTAs ("Browse workspace", "Start a session").
- Disabled states: Submit button disables when loading or token is empty (LoginPrompt.tsx:223). Input disables during loading (line 212).
- Confirmation for destructive actions: Archive and delete dialogs use ConfirmDialog with `destructive` prop (SessionHeader.tsx:323, 336).
- Reduced motion: login.css respects `prefers-reduced-motion: reduce` (line 642-649) with zero-duration animations.
- ContextMenu keyboard navigation: ArrowUp/Down wraps, Home/End supported, Escape closes, Tab closes (ContextMenu.tsx:70-103).
- `prefers-reduced-motion` zeroes ALL animation and transition durations (login.css:642-649).
- `aria-busy={isLoading}` on submit button (LoginPrompt.tsx:224).

**Issues found:**
- ContextMenu `minHeight: 36` fails the 44px mobile touch target standard (ContextMenu.tsx:143). On touch devices, menu items are too small to tap accurately.
- No ErrorBoundary wrapping for the login page -- if the login component throws during render (e.g., API client failure), the user sees a blank screen with no recovery path.
- The `useEffect` cleanup in LoginPrompt (line 70) deletes `document.body.dataset.loginActive` on unmount, but if the component errors during mount, the dataset attribute persists, potentially leaving stale state.
- SessionHeader has no loading state for the session title -- if session metadata loads slowly, the truncated title displays `session.id.slice(0, 8)` which is cryptic.
- Mobile form panel `justify-content: flex-start` (login.css:546) with `padding-top: 0` means the form starts immediately after the mobile brand section with no breathing room if the mobile viewport is short.
