## Top 3 Priority Fixes

1. **Login page ignores UI-SPEC split-screen layout** -- UI-SPEC-login.md specifies a 45/55 split layout with warm clay gradient left panel, decorative terminal ASCII art, and staggered entry animations. The implemented login uses a single-column centered layout with no gradient panel or brand visual. This is the single largest design contract deviation. **Fix:** Implement the split layout with the clay accent gradient left panel and the decorative terminal element as specified in UI-SPEC sections 4, 5.1.

2. **11 error text instances use hardcoded `text-red-*` Tailwind classes instead of `--hp-danger` design token** -- Files: `router.tsx:220`, `App.tsx:410`, `SessionChat.tsx:528`, `RequestUserInputFooter.tsx:203`, `_results.tsx:656/781/896`, `AttachmentItem.tsx:42/46/47`, `ToolMessage.tsx:284`. These bypass the design token system and will not adapt to dark mode. **Fix:** Replace all `text-red-500` / `text-red-600` with `text-[var(--hp-danger)]`.

3. **2 hardcoded English strings bypass i18n system** -- `AttachmentItem.tsx:47` has `"Upload failed"` and `routes/sessions/file.tsx:270` has `"Save"` as raw English strings. Non-English users will see untranslated text. **Fix:** Add i18n keys and use `t('...')` for these strings.

## Detailed Findings

### Pillar 1: Copywriting (2/4)

**Findings:**
- `web/src/components/AssistantChat/AttachmentItem.tsx:47` -- Hardcoded English string `"Upload failed"` not wrapped in i18n `t()`.
- `web/src/routes/sessions/file.tsx:270` -- Hardcoded `"Save"` button label not wrapped in i18n.
- `web/src/components/SessionList.tsx:36-38` -- Empty state has hardcoded English strings: `"Clone a project"`, `"Edit & Develop"`, `"Review & Push"` and their descriptions. These should use `t()` keys.
- `web/src/components/AgentFlavorIcon.tsx:3-26` -- Agent labels (`'Cl'`, `'Cx'`, `'Cu'`, `'Gm'`, `'Km'`, `'Op'`) are hardcoded abbreviations, though these are acceptable as brand identifiers.
- Positive: Login page correctly uses `t()` for all user-visible strings including errors, buttons, and footer.
- Positive: Dialog confirmations use `t()` with parameterized session names.
- Positive: `LoadingState` component uses `t('loading')` fallback.

### Pillar 2: Visuals (2/4)

**Findings:**
- **BLOCKER: Login layout diverges from UI-SPEC-login.md.** The spec calls for:
  - Split-screen layout (45/55) with warm clay gradient left panel
  - Decorative terminal ASCII art (`> hapi-power --login connecting...`) with blinking cursor
  - Brand name on gradient panel with text shadow
  - Multiple background layers (gradient + noise + geometric pattern + radial glow)
  
  The actual implementation (`login.css`) uses:
  - Single-column centered layout (`.login-page { display: flex; flex-direction: column; align-items: center }`)
  - No gradient panel, no decorative terminal element
  - Brand name centered with `text-align: center`, no text shadow
  - Background is flat `var(--hp-canvas)` instead of the specified multi-layer gradient

- Login page animations present and using design tokens (`--hp-duration-*`, `--hp-ease-*`).
- Session list has proper visual hierarchy: machine > project group > session item with 3-level collapsible tree.
- Settings page uses consistent card-based layout with proper section headers.
- All pages use consistent `max-w-content` for horizontal containment.
- Icon-only buttons have `title` attributes (e.g., `title={t('sessions.new')}`).
- Focus-visible rings present on interactive elements (`focus-visible:ring-2 focus-visible:ring-[--hp-primary]`).

### Pillar 3: Color (2/4)

**Findings:**
- **11 instances of hardcoded `text-red-*`** that should use `--hp-danger`:
  - `router.tsx:220` -- `text-red-600` for session error
  - `App.tsx:410` -- `text-red-600` for auth error
  - `SessionChat.tsx:528` -- `text-red-600` for chat error
  - `RequestUserInputFooter.tsx:203` -- `text-red-600`
  - `_results.tsx:656,781,896` -- `text-red-600` for tool result errors
  - `AttachmentItem.tsx:42,46,47` -- `text-red-500` for upload errors
  - `ToolMessage.tsx:284` -- `text-red-500` for error label

- **6 hardcoded hex colors in `AgentFlavorIcon.tsx`** (lines 4-24): `#d97706`, `#111827`, `#0f766e`, `#2563eb`, `#7c3aed`, `#15803d`. These are brand-identity colors for agent types -- acceptable for badge colors but should ideally use design tokens or a defined palette.

- **15+ hardcoded hex colors in `FileIcon.tsx`** (lines 4-19): File-type colors (`#3178c6` for TypeScript, `#f7df1e` for JS, etc.). These are conventional language colors and are acceptable.

- **Mermaid diagram hardcoded colors** in `mermaid-diagram.tsx:30-35`: `#323843`, `#edf1f5`, `#6d8fd6`, `#94a3b8`, `#2d3440`, `#2a2f35` -- these are dark-theme only colors that will not adapt to light mode.

- **Rainbow animation hardcoded colors** in `index.css:427-434` -- intentional decorative effect, acceptable.

- Positive: Design token system (`tokens.css`) is comprehensive with oklch-based colors for both light and dark themes.
- Positive: `--app-*` semantic alias layer correctly references `--hp-*` tokens with Telegram fallbacks.
- Positive: `login.css` correctly uses `--hp-*` design tokens throughout.
- Positive: Dark mode variables are properly defined in `[data-theme="dark"]` selector.

### Pillar 4: Typography (3/4)

**Findings:**
- Font stack correctly declared: DM Sans (UI), Source Serif 4 (Display), JetBrains Mono (Code) per `typography.css`.
- Source Serif 4 used only in login brand name and dialog title (appropriate display usage).
- DM Sans used as the body font via `--hp-font-sans`.
- 5 Tailwind font sizes in use: `text-xs`, `text-sm`, `text-base`, `text-xl`, `text-2xl`, `text-3xl`.
- Custom pixel sizes: `text-[10px]`, `text-[11px]` appear in multiple files (DiffView, FileIcon, CodeBlock, SessionList). These bypass the type scale.
- 3 font weights used: `font-medium` (most common), `font-semibold`, `font-bold`. This is within reasonable range.
- `typography.css` defines responsive type scale via `clamp()` which is good practice.
- Login page uses `font-weight: 700` for brand name vs UI-SPEC specifying `600` -- minor deviation.
- Code editor fallback uses hardcoded monospace stack (`ui-monospace, SFMono-Regular...`) instead of `var(--hp-font-mono)` in `index.css:400`.

### Pillar 5: Spacing (3/4)

**Findings:**
- Comprehensive spacing scale defined in `tokens.css`: `--hp-space-0_5` (2px) through `--hp-space-12` (48px).
- Login page correctly uses spacing tokens in CSS (e.g., `margin-top: 12px` maps to space-3, `padding: 0 16px` maps to space-4).
- Tailwind spacing classes (`p-3`, `px-4`, `gap-2`, etc.) used extensively alongside `var(--hp-space-*)` -- dual system creates inconsistency risk.
- Arbitrary spacing values found:
  - `text-[10px]`, `text-[11px]` -- font sizes outside the type scale
  - `max-w-[140px]`, `max-w-[150px]`, `min-w-[160px]` -- component-specific widths
  - `min-h-[44px]`, `min-h-[32px]`, `min-h-[36px]` -- touch target sizes (44px is correct per spec)
  - `h-[28px]` in UpdateBanner
- `SubPageLayout.tsx` uses inline styles with `var(--hp-space-*)` instead of Tailwind classes -- consistent with token system but inconsistent with the Tailwind approach used elsewhere.
- Border-radius inconsistency: Mixed use of `rounded-lg`, `rounded-md`, `rounded-xl`, `rounded-[var(--hp-radius-sm)]`, `rounded-[var(--hp-radius-md)]`. The design tokens define `--hp-radius-sm: 6px` and `--hp-radius-md: 10px`, but Tailwind's `rounded-lg` is 8px and `rounded-md` is 6px. This means `rounded-lg` (8px) is used where `--hp-radius-md` (10px) should be.

### Pillar 6: Experience Design (3/4)

**Findings:**
- **Loading states:** Present across all data-fetching views:
  - `LoadingState` component with spinner and skeleton variants
  - `skeleton-shimmer` CSS animation for placeholder loading
  - Login button shows spinner + disabled state during submission
  - New session form disables during creation
  
- **Error states:** Present but inconsistent:
  - Login page has proper error display with `role="alert"`, `--hp-danger` colors, and styled error container
  - Session list has error display but uses hardcoded `text-red-600`
  - Chat page has error display
  - New session form shows spawn errors
  - No global React ErrorBoundary found

- **Empty states:** Well-handled:
  - `SessionsEmptyState` component with steps guide and action buttons
  - Search no-results message
  - Git changes empty state with messaging
  - Extensions empty states

- **Confirmation dialogs:** Present for destructive actions (archive, delete) with `ConfirmDialog` component.

- **Accessibility:** 64 aria attributes found across components. `role="alert"` on error containers. `aria-busy` on loading buttons. Keyboard navigation supported.

- **Reduced motion:** Two `prefers-reduced-motion` media queries:
  - `login.css:435-442` -- disables all animations on login page
  - `index.css:579-585` -- global reduced motion support

- **Mobile responsiveness:** 
  - Login page has responsive breakpoints at 768px and 400px
  - Session header uses `sm:` and `lg:` breakpoints
  - iOS zoom prevention present (`font-size: max(1rem, 16px) !important`)
  - Safe area insets handled (`env(safe-area-inset-*)`)

- Missing: Toast/notification system for transient errors (toast-context exists but limited usage).
- Missing: No offline/slow-network indicator beyond `OfflineBanner` component.

## Login Page: UI-SPEC Deviation Summary

The login page implementation diverges significantly from `UI-SPEC-login.md` in these areas:

| Spec Requirement | Status | Notes |
|---|---|---|
| Split-screen layout (45/55) | MISSING | Single-column centered layout instead |
| Warm clay gradient left panel | MISSING | Flat `--hp-canvas` background |
| Decorative terminal ASCII art | MISSING | Not implemented |
| Brand name with text shadow | PARTIAL | Brand name present, no text shadow |
| Multi-layer background (noise, pattern, glow) | MISSING | No background layers |
| Staggered page-load animation | PARTIAL | Fade/slide animations present but simpler than spec |
| "ACCESS TOKEN" uppercase label | MISSING | Uses regular-case label from i18n |
| Input with 8px border-radius | DEVIATED | Uses `--hp-radius-md` (10px) |
| Hub URL dialog | PRESENT | Correctly implemented with dialog component |
| Language switcher | PRESENT | Correctly positioned |
| Dark mode support | PRESENT | Token system handles dark mode |
| Reduced motion | PRESENT | `prefers-reduced-motion` respected |
| Mobile responsive | PRESENT | Breakpoints at 768px and 400px |
