# Hapi Power i18n 国际化方案研究

> 研究日期：2026-05-30
> 目标：为 Hapi Power 设计中英双语国际化方案
> 技术栈：React 19 + Vite + Bun + Hono + TypeScript monorepo

---

## 一、现状分析

### 1.1 当前 i18n 实现（双轨并行）

项目当前存在 **两套独立的 i18n 实现**：

#### web/ — 自研轻量方案（已投入生产）

- **文件位置**：`web/src/lib/i18n-context.tsx` + `web/src/lib/use-translation.ts`
- **翻译文件**：`web/src/lib/locales/en.ts` / `zh-CN.ts`（TypeScript 对象，约 450 行/文件）
- **实现方式**：自研 `I18nProvider` + `useTranslation` hook
- **变量插值**：`{key}` 格式，正则替换
- **持久化**：`localStorage.setItem('hapi-lang', locale)`
- **回退策略**：当前 locale -> en -> key 本身
- **使用范围**：54 个文件，125 处引用，约 200+ 个翻译 key
- **优点**：零依赖，bundle size 极小，TypeScript 类型安全（`as const`）
- **缺点**：无命名空间，无懒加载，无复数/日期格式化，无自动提取工具

#### website/ — react-i18next 方案（已安装未充分使用）

- **文件位置**：`website/src/i18n.ts`
- **翻译文件**：`website/src/locales/en.json` / `zh.json`（嵌套 JSON，约 130 行/文件）
- **依赖**：`i18next@25.7.3` + `react-i18next@16.5.0` + `i18next-browser-languagedetector@8.2.0`
- **使用范围**：6 个文件，11 处引用
- **变量插值**：`{{key}}` 格式
- **特点**：嵌套 JSON 结构（按页面/组件组织），使用 `i18next-browser-languagedetector` 自动检测

### 1.2 两套方案的不一致问题

| 维度 | web/ 自研 | website/ react-i18next |
|------|----------|----------------------|
| 翻译格式 | TypeScript 对象（`.ts`） | JSON（`.json`） |
| key 结构 | 扁平化 `section.subsection.key` | 嵌套 `section.subsection.key` |
| 变量语法 | `{key}` | `{{key}}` |
| 加载方式 | 全量 import | 全量 import |
| 依赖 | 零 | i18next 全家桶 |

## 二、i18n 库评估

### 2.1 候选方案对比

| 维度 | react-i18next (i18next) | Lingui | FormatJS (react-intl) | 自研轻量 |
|------|------------------------|--------|----------------------|---------|
| Bundle size (gzip) | ~40KB (i18next 12KB + react-i18next 6KB + detector 3KB) | ~15KB | ~30KB | ~2KB |
| TypeScript 支持 | 良好（key 类型需额外配置） | 优秀（ICU message） | 良好 | 优秀（原生 TS） |
| React 19 支持 | v16+ 支持 | v5+ 支持 | v6+ 支持 | 原生支持 |
| 懒加载 | 支持（namespace + backend plugin） | 支持（按需编译） | 支持（@formatjs/fast-bundle） | 需自研 |
| 复数/性别/选择 | 支持（ICU via plugin） | 原生 ICU | 原生 ICU | 需自研 |
| 日期/数字格式化 | 不内置，需 Intl API 或 date-fns | 不内置 | 内置（Intl） | 需 Intl API |
| 自动提取 | i18next-parser | lingui extract | @formatjs/cli | 需自研 |
| 翻译缺失处理 | 回退链 + ns | 回退 + 编译时检查 | 编译时检查 | 运行时回退 |
| 生态成熟度 | 最高（star 最多） | 中等 | 中等 | N/A |
| SSR 支持 | 支持 | 支持 | 支持 | 需自研 |

### 2.2 项目实际情况考量

1. **只需要中英双语**（LTR），不需要 RTL 支持
2. **web/ 已经有 200+ key 的自研方案在运行**，54 个文件在使用
3. **website/ 已安装 react-i18next**，但使用量很小
4. **cli/ 和 hub/（Bun 后端）** 目前无 i18n 需求
5. **日期格式化**：项目已有 `date-fns` 依赖（间接引入）
6. **不需要 ICU MessageFormat** 的复杂功能（复数、性别等）
7. **Bundle size 敏感**：web/ 是 PWA，需要最小化体积

### 2.3 评估结论

**对 Hapi Power 的场景，自研轻量方案是最优解。**

理由：

1. **需求简单**：只有中英双语，无 ICU 复数/性别，无 RTL
2. **已有投入**：web/ 已经有成熟的自研方案运行，54 个文件 125 处调用
3. **迁移成本**：从自研切换到 react-i18next 需要改写所有调用点，收益不大
4. **Bundle size**：自研方案 ~2KB vs react-i18next ~40KB（省 ~38KB）
5. **类型安全**：TypeScript 对象天然有类型推导，比 JSON key 字符串更安全
6. **统一性**：website/ 也应统一到自研方案，消除双轨不一致

## 三、推荐方案：统一的自研轻量 i18n

### 3.1 架构设计

```

  ├── types.ts                # Locale, TranslationKey, TranslateFn 类型

  ├── use-translation.ts      # useTranslation hook

  └── locales/
```

### 3.2 核心类型定义

```typescript
// shared/src/i18n/types.ts

export type Locale = 'en' | 'zh-CN'

// 翻译字典类型 — 使用 flat key-value
export type TranslationDict = Record<string, string>

// 翻译函数签名
export type TranslateFn = (
  key: string,
  params?: Record<string, string | number>
) => string

// Context 值类型
export interface I18nContextValue {
  t: TranslateFn
  locale: Locale
  setLocale: (locale: Locale) => void
  formatDate: (date: Date | number | string, options?: Intl.DateTimeFormatOptions) => string
  formatNumber: (value: number, options?: Intl.NumberFormatOptions) => string
}

// 支持的 locale 列表（用于 UI 渲染）
export const SUPPORTED_LOCALES: { value: Locale; label: string }[] = [
  { value: 'en', label: 'English' },
  { value: 'zh-CN', label: '简体中文' },
]
```

### 3.3 Provider 实现

```typescript
// shared/src/i18n/context.tsx
import { createContext, useEffect, useState, useCallback, type ReactNode } from 'react'
import type { Locale, TranslationDict, I18nContextValue } from './types'
import { interpolate, createDateFormatter, createNumberFormatter } from './format'

export type { Locale, I18nContextValue } from './types'
export const I18nContext = createContext<I18nContextValue | null>(null)

const STORAGE_KEY = 'hapi-lang'

// 延迟导入翻译字典，未来支持按 namespace 懒加载
import { en } from './locales/en'
import { zhCN } from './locales/zh-CN'

const locales: Record<Locale, TranslationDict> = { en, 'zh-CN': zhCN }

function detectLocale(): Locale {
  const saved = localStorage.getItem(STORAGE_KEY)
  if (saved === 'en' || saved === 'zh-CN') return saved

  // 浏览器语言检测
  const browserLang = navigator.language
  if (browserLang.startsWith('zh')) return 'zh-CN'

  return 'en'
}

export function I18nProvider({ children }: { children: ReactNode }) {
  const [locale, setLocaleState] = useState<Locale>(detectLocale)

  const setLocale = useCallback((newLocale: Locale) => {
    setLocaleState(newLocale)
    localStorage.setItem(STORAGE_KEY, newLocale)
    document.documentElement.lang = newLocale
  }, [])

  const t = useCallback((key: string, params?: Record<string, string | number>): string => {
    const dict = locales[locale] ?? locales.en
    const value = dict[key]
    const fallback = locales.en[key] ?? key
    return interpolate(value ?? fallback, params)
  }, [locale])

  const formatDate = createDateFormatter(() => locale)
  const formatNumber = createNumberFormatter(() => locale)

  useEffect(() => {
    document.documentElement.lang = locale
  }, [locale])

  return (
    <I18nContext.Provider value={{ t, locale, setLocale, formatDate, formatNumber }}>
      {children}
    </I18nContext.Provider>
  )
}
```

### 3.4 格式化工具

```typescript
// shared/src/i18n/format.ts

/** 变量插值：{key} -> value */
export function interpolate(
  str: string,
  params?: Record<string, string | number>
): string {
  if (!params) return str
  return str.replace(/\{(\w+)\}/g, (match, key) => {
    const value = params[key]
    return value !== undefined ? String(value) : match
  })
}

/** 日期格式化工厂 */
export function createDateFormatter(
  getLocale: () => string
): (date: Date | number | string, options?: Intl.DateTimeFormatOptions) => string {
  return (date, options) => {
    const d = typeof date === 'string' ? new Date(date) : new Date(date)
    return new Intl.DateTimeFormat(getLocale(), {
      dateStyle: 'medium',
      timeStyle: 'short',
      ...options,
    }).format(d)
  }
}

/** 数字格式化工厂 */
export function createNumberFormatter(
  getLocale: () => string
): (value: number, options?: Intl.NumberFormatOptions) => string {
  return (value, options) => {
    return new Intl.NumberFormat(getLocale(), options).format(value)
  }
}
```

### 3.6 翻译文件组织

**原则：扁平化 key + 模块注释分组，不使用嵌套对象**

```typescript
// shared/src/i18n/locales/en.ts
export const en: Record<string, string> = {
  // ======== Loading states ========
  'loading': 'Loading...',
  'loading.session': 'Loading session...',
  'loading.git': 'Loading Git status...',

  // ======== Login / Auth ========
  'login.title': 'HAPI',
  'login.submit': 'Sign In',
  'login.error.authFailed': 'Authentication failed',

  // ======== Sessions ========
  'sessions.count': '{n} sessions in {m} projects',
  'sessions.new': 'New Session',

  // ======== Files ========
  'files.page.title': 'Files',
  'files.changes.error.gitStatusUnavailable': 'Git status unavailable.',

  // ... (保持现有的扁平化 key 结构)
} as const
```

**为什么选择扁平化 key 而非嵌套 JSON**：

1. TypeScript `as const` 直接推导 key 类型，嵌套对象推导复杂
2. 搜索/替换方便：`Ctrl+F 'login.error'` 一键定位
3. 避免嵌套层级误操作：`sessions.count` 不会被误写成 `session.count`
4. i18next-parser 和 lingui extract 生成的 key 也是扁平化的
5. 当前 web/ 已在使用此模式，无需迁移

### 3.7 website/ 迁移：从 react-i18next 到自研

website/ 当前使用嵌套 JSON + react-i18next，迁移步骤：

1. 将 `website/src/locales/zh.json` 嵌套结构展平为扁平 key
2. 将 `.json` 转为 `.ts`（保留类型安全）
3. 删除 `i18next`、`react-i18next`、`i18next-browser-languagedetector` 依赖
4. 将 `useTranslation` 调用从 `const { t } = useTranslation()` 迁移到自研版本
5. 将 `t('hero.title')` 改为 `t('hero.title')`（key 格式基本不变，只是变量从 `{{key}}` 改为 `{key}`）

## 五、翻译工作流

### 5.1 提取流程

由于使用 TypeScript 对象而非 JSON，不需要额外的提取工具。翻译 key 就是 TypeScript 的属性名：

1. **开发者**在组件中直接写 `t('feature.label')`
2. **TypeScript 编译器**自动检查 key 是否存在于 `en.ts`（通过 `as const` + 类型工具）
3. **翻译者**只需对照 `en.ts` 在 `zh-CN.ts` 中填写对应翻译

### 5.2 翻译缺失检测

```typescript
// shared/src/i18n/locales/validate.ts
// 开发时可选运行，检查 zh-CN 是否覆盖所有 en 的 key

import { en } from './en'
import { zhCN } from './zh-CN'

export function findMissingTranslations(): string[] {
  const enKeys = Object.keys(en)
  const zhKeys = new Set(Object.keys(zhCN))
  return enKeys.filter(key => !zhKeys.has(key))
}

// vitest 测试
if (import.meta.vitest) {
  const { it, expect } = import.meta.vitest
  it('zh-CN covers all en keys', () => {
    const missing = findMissingTranslations()
    expect(missing).toEqual([])
  })
}
```

### 5.3 Fallback 策略

```
当前 locale 查找 → en 基准查找 → 返回 key 本身
```

已存在于当前实现中，无需修改。

### 5.4 翻译质量保证

- `en.ts` 是 **Source of Truth**，所有新 key 必须先在 en.ts 中定义
- `zh-CN.ts` 必须覆盖所有 en.ts 的 key（通过测试保证）
- CI 中运行翻译完整性测试

## 六、日期/数字本地化

### 6.1 使用 Intl API（零依赖）

项目不需要重量级的日期格式化库。Intl API 是浏览器原生支持的：

```typescript
// 日期格式化
formatDate(new Date(), { dateStyle: 'medium' })
// en: "May 30, 2026"
// zh-CN: "2026年5月30日"

// 相对时间（"3 分钟前"）
const rtf = new Intl.RelativeTimeFormat(locale, { numeric: 'auto' })
rtf.format(-3, 'minute')
// en: "3 minutes ago"
// zh-CN: "3分钟前"

// 数字格式化
formatNumber(1234567)
// en: "1,234,567"
// zh-CN: "1,234,567"
```

### 6.2 相对时间工具

当前项目已有 `session.time.minutesAgo` / `session.time.hoursAgo` / `session.time.daysAgo` 等翻译 key。
可以直接使用 `Intl.RelativeTimeFormat` 替代硬编码的时间 key：

```typescript
// shared/src/i18n/format.ts
export function formatRelativeTime(
  date: Date | number,
  getLocale: () => string
): string {
  const now = Date.now()
  const diff = (typeof date === 'number' ? date : date.getTime()) - now
  const absDiff = Math.abs(diff)

  const rtf = new Intl.RelativeTimeFormat(getLocale(), { numeric: 'auto' })

  if (absDiff < 60_000) return rtf.format(0, 'second')
  if (absDiff < 3_600_000) return rtf.format(-Math.floor(absDiff / 60_000), 'minute')
  if (absDiff < 86_400_000) return rtf.format(-Math.floor(absDiff / 3_600_000), 'hour')
  return rtf.format(-Math.floor(absDiff / 86_400_000), 'day')
}
```

## 七、RTL 支持

### 7.1 当前不需要

中文和英文都是 LTR（Left-to-Right）语言，RTL 支持在当前阶段不是必需的。

### 7.2 预留方案

如果未来需要支持阿拉伯语（ar）或希伯来语（he）：

1. `document.documentElement.dir = isRTL(locale) ? 'rtl' : 'ltr'`
2. CSS 使用 `margin-inline-start` 替代 `margin-left`，`padding-inline-end` 替代 `padding-right`
3. Tailwind v4 的 RTL 支持：`ms-*` / `me-*` / `ps-*` / `pe-*` 等逻辑属性

```typescript
const RTL_LOCALES = new Set(['ar', 'he', 'fa', 'ur'])
export function isRTL(locale: string): boolean {
  return RTL_LOCALES.has(locale.split('-')[0])
}
```

这个改动量很小，不需要现在做，但架构上已预留位置。

## 九、服务端翻译（Bun 后端）

### 9.1 hub/ 后端是否需要 i18n？

分析 hub/ 的场景：

- **API 错误消息**：当前返回英文错误消息，面向开发者，不需要 i18n
- **Telegram Bot 消息**：通知消息需要根据用户语言设置返回对应语言
- **WebSocket 推送**：数据推送，不涉及翻译

**结论**：hub/ 后端需要 **有限度的 i18n**，主要用于 Telegram Bot 通知消息。

### 9.2 服务端 i18n 方案

不引入 react-i18next（服务端无 React），直接复用 shared 的翻译字典：

```typescript
// hub/src/i18n.ts
import { en, zhCN } from '@hapi/shared/i18n/locales'  // 或直接 import
import type { Locale } from '@hapi/shared/i18n/types'

const locales = { en, 'zh-CN': zhCN } as const

export function t(
  locale: Locale,
  key: string,
  params?: Record<string, string | number>
): string {
  const dict = locales[locale] ?? locales.en
  const template = dict[key] ?? locales.en[key] ?? key
  if (!params) return template
  return template.replace(/\{(\w+)\}/g, (_, k) =>
    params[k] !== undefined ? String(params[k]) : `{${k}}`
  )
}
```

使用示例（Telegram Bot）：

```typescript
// hub/src/telegram/handlers.ts
ctx.reply(t(userLocale, 'toast.task.completed'))
ctx.reply(t(userLocale, 'toast.ready.body', {
  agent: session.agent,
  session: session.name,
}))
```

### 9.3 CLI 端

cli/ 使用 `ink`（React for CLI），理论上可以复用 `I18nProvider`，
但 CLI 工具通常不需要 i18n（终端用户默认英文），暂不实现。

4. 更新 `web/` 中硬编码的相对时间（如 `session.time.minutesAgo`）改为使用 `Intl.RelativeTimeFormat`

**预估工作量**：2-3 小时

### 10.3 Phase 3：website/ 迁移（优先级：中）

**目标**：统一 website/ 的 i18n 方案

**步骤**：

1. 将 `website/src/locales/zh.json` 嵌套结构展平为 `zh-CN.ts` 扁平格式
2. 将 `website/src/locales/en.json` 转为 `en.ts` 扁平格式

## 十二、不建议采用 react-i18next 的原因总结

1. **已有成熟的自研方案在运行**，无迁移理由
2. **需求简单**（中英双语，无 ICU），react-i18next 的能力过剩
3. **Bundle size**：自研 ~2KB vs react-i18next ~40KB
4. **类型安全**：TypeScript 对象天然有类型推导，JSON + 字符串 key 的方案反而更弱
5. **迁移成本**：需改写 54 个文件 125 处调用，收益为零
6. **维护成本**：react-i18next 大版本升级可能有 breaking change，自研方案完全可控

如果未来需求变化（如需要 10+ 语言、需要 ICU 复数/性别、需要专业翻译工具集成），
再考虑迁移到 react-i18next。当前阶段自研方案是最务实的选择。
