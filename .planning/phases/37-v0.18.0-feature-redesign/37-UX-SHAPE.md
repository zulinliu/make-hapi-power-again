# Phase 37 — 五大特色功能 UX Shape

> 优先级：前端效果 > 交互闭环 > 状态反馈 > API 支撑
> 原则：移动端不是缩小版桌面端；所有主操作触控目标不小于 44px；默认动画 150–250ms；支持 `prefers-reduced-motion`。

## 0. 统一体验原则

1. **驾驶舱而非表格**：每个功能先给用户一个“当前状态 + 推荐动作”，再展开细节。
2. **状态显性**：在线/失败、已排队/已引导、上下文风险、Git 同步方向都要直接可见。
3. **动作有后果提示**：中断、强推、删除供应商、导出含敏感内容等动作必须有清晰 microcopy。
4. **移动端底部动作优先**：iOS 单手操作依赖底部 action dock / sheet，不把核心动作藏在右上角。
5. **不靠颜色唯一表达**：上下文颜色同时配合文字、点状风险、tooltip 说明。
6. **低装饰、高质感**：用布局、留白、信息层级和轻动效提升高级感，不使用重渐变、花哨阴影。

## 1. API 星桥 / Model Nexus

### 1.1 当前体验问题

- Provider 位于 Settings 中的一个普通区块，视觉权重不足。
- 用户看不到“哪个 provider 正在被哪个 agent 使用”。
- 模型发现只反馈数量，不告诉失败原因、协议兼容性、usage/context 支持。
- 明文 API Key 读取接口存在，UI 不应鼓励把密钥当普通字段管理。
- 新增流程是空表单，不会引导用户选择协议、测试连接、分配 agent。

### 1.2 新页面结构

建议在设置页中保留入口，同时把内容升级为独立感强的 **API 星桥控制舱**。

```text
┌──────────────────────────────────────────────┐
│ API 星桥 Model Nexus                         │
│ 连接模型通道，并为每个 Agent 分配默认能力     │
├──────────────────────────────────────────────┤
│ 健康总览：3 在线 · 1 待检测 · 0 高风险        │
│ [新增星桥] [一键检测] [安全策略]              │
├───────────────┬──────────────────────────────┤
│ Provider 星图  │ Agent 分配矩阵               │
│ - Tsintergy ●  │ Claude   glm-5.1  默认        │
│ - OpenAI    ●  │ Codex    gpt-5-codex         │
│ - Gemini    ◐  │ Gemini   gemini-...           │
├───────────────┴──────────────────────────────┤
│ 选中 Provider 详情：模型、延迟、用量支持、日志 │
└──────────────────────────────────────────────┘
```

### 1.3 核心模块

#### A. 顶部健康总览

- **在线**：最近检测成功且模型列表可用。
- **待检测**：从未检测或配置刚变更。
- **风险**：HTTP、证书、usage 不支持、base URL 不可信、模型发现失败。
- 视觉：四个小卡片，紧凑但有明确数字；激活色使用 Electric Orange，状态色使用 success/warning/danger tokens。

#### B. Provider 星图卡片

每个 Provider 是一个“节点卡”：

- 名称 + base host（不显示完整 token）。
- 协议标签：`Anthropic-compatible` / `OpenAI-compatible` / `Gemini` / `Auto`。
- 健康点：在线、检测中、失败、部分能力缺失。
- Agent badges：Claude、Codex、Gemini 等。
- 模型摘要：`12 models · usage ✓ · ctx ?`。
- 最近检测：`2 分钟前`。

交互：

- 点击卡片进入右侧/下方详情。
- 长按/右键打开操作菜单：检测、编辑、复制 Base URL、删除。
- 失败卡片默认展开“修复建议”，例如鉴权失败、404、协议不匹配。

#### C. Agent 分配矩阵

替代当前 chip：

| Agent | 默认 Provider | 默认模型 | 备用 Provider | 状态 |
|---|---|---|---|---|
| Claude | Tsintergy | glm-5.1 | Anthropic | Usage ✓ |
| Codex | OpenAI | gpt-5-codex | - | Ready |
| Gemini | Google | gemini-... | - | Need key |

- 单元格点击弹出 provider/model picker。
- 默认模型 picker 按 provider 分组，显示延迟、context、usage 支持。
- 没有配置时显示 `为 Claude 接入第一个模型通道`，不是空表。

#### D. 新增星桥 Wizard

四步，不再一次性空表单：

1. **选择协议**：Anthropic-compatible / OpenAI-compatible / Gemini / Auto detect。
2. **输入连接**：名称、Base URL、API Key；默认隐藏 key；HTTP 明示风险。
3. **测试能力**：连通性、模型发现、用量字段、context window 推断。
4. **分配 Agent**：设为 Claude 默认 / 仅备用 / 暂不分配。

Wizard 完成后显示“星桥点亮”动效：节点亮起，桥线扫光 160ms。

#### E. 诊断抽屉

每次检测记录：

- 请求候选 endpoint：`/v1/models`、strip suffix 等只显示 URL host/path，不显示 key。
- HTTP 状态、耗时、模型数量。
- usage test：是否返回 `input_tokens/output_tokens/context_tokens/context_window`。
- 安全提示：HTTP、内网、重定向、证书、DNS。

### 1.4 移动端设计

- 顶部横向健康 chips，可左右滑。
- Provider 星图改为纵向卡片；点击打开全屏 bottom sheet。
- Agent 分配矩阵改为“Agent 卡片列表”，每个卡片底部有 `更换模型` 按钮。
- Wizard 使用 stepper sheet，底部固定 `下一步`。
- 所有表单输入保持 16px 以上，避免 iOS 自动缩放。

### 1.5 动效

| 场景 | 动效 |
|---|---|
| 检测中 | 节点外圈 1.4s 慢速脉冲，减少动画时改为静态 spinner。 |
| 检测成功 | 桥线从左到右扫光一次，节点变 success。 |
| 检测失败 | 卡片轻微 horizontal shake 120ms + 展开错误原因；不使用长时间红色闪烁。 |
| 分配 Agent | 模型 badge 从 Provider 卡片飞入 Agent 单元格，120ms scale/fade。 |

## 2. Git 脉络 / Git Atlas

### 2.1 当前体验问题

- Tab 结构按命令拆分，用户需要自己判断下一步。
- 状态页只列文件，没有 staged/commit basket/agent turn 分组。
- toolbar 操作平铺，缺少当前分支、远端同步方向、风险提示。
- 移动端 Git 入口藏在 More；页面本身仍偏桌面表格。
- 前端解析 command output，导致 UI 状态难统一。

### 2.2 新页面信息架构

Git Atlas 应是“会话内代码变更地图”。默认首屏回答三个问题：

1. 我在哪条分支？
2. Agent 改了什么？
3. 现在最安全的动作是什么？

桌面布局：

```text
┌──────────────────────────────────────────────┐
│ Git 脉络 · branch feat/v0.17.3  ↑2 ↓0         │
│ 推荐动作：[检查 12 个变更] [提交篮] [推送]     │
├───────────────┬──────────────────────────────┤
│ 变更地图       │ Diff / 文件预览               │
│ Agent turn A   │                              │
│  M web/...     │                              │
│  A hub/...     │                              │
│ 未分组变更      │                              │
├───────────────┴──────────────────────────────┤
│ 时间线 · 分支 · 远端 · 克隆/同步               │
└──────────────────────────────────────────────┘
```

移动布局：

```text
[Git 脉络]
branch feat/v0.17.3    ↑2 ↓0
工作区：12 个变更
[检查变更]
[提交篮 0/12]
[同步远端]

卡片：Agent Turn 12: Fix upload
  5 files changed
  [查看] [加入提交篮]

底部 dock: 状态 · 变更 · 提交 · 同步
```

### 2.3 核心模块

#### A. 分支态势 Hero

- 当前分支、上游、ahead/behind、dirty count、冲突 count。
- “推荐动作”根据状态生成：
  - clean + behind：`拉取更新`
  - dirty：`检查变更`
  - dirty + selected files：`提交所选`
  - ahead：`推送`
  - conflict：`处理冲突`
- Hero 背景用极浅橙色几何线，不做重渐变。

#### B. 变更地图

- 文件按来源分组：
  1. 最近 agent turn（如果能从 message/change tracking 关联）。
  2. 用户手动改动。
  3. 未追踪文件。
  4. 冲突文件。
- 每个文件行显示：状态 badge、路径、行数变更、是否已加入提交篮。
- 点击文件默认打开右侧 diff preview；长按/更多：复制路径、打开文件、加入提交篮、忽略。

#### C. 提交篮 Commit Basket

- 替代“提交弹窗里勾文件”的被动模式。
- 用户在变更地图中把文件加入篮子；底部/右侧显示 `提交篮 4/12`。
- Basket 内提供：提交消息建议、按 agent turn 生成 commit message、预览最终文件列表。
- 没有 staging API 时，第一版仍可通过现有 `git commit` 选文件参数实现；UI 不必须暴露 Git staging 概念。

#### D. Diff 预览

- 桌面右侧 sticky pane。
- 移动端 bottom sheet，全屏可左右切换文件。
- 大文件或二进制文件显示摘要和打开文件入口。
- 冲突文件提供“打开文件编辑”主动作。

#### E. 同步中心

- Fetch/Pull/Push 变为一个“同步中心”sheet：
  - 显示 remote、upstream、ahead/behind。
  - 普通 Push/Pull 主按钮。
  - Force push 收进危险区，需输入分支名确认。
- 同步操作使用箭头流向动效，完成后在 Hero 上更新计数。

#### F. 时间线 / 分支地图

- 历史不再是单纯列表，改为轻量 commit timeline。
- 每个 commit 点可展开：hash、作者、时间、message、文件摘要。
- Branch manager 通过“切换/新建/合并”动作面板进入，不占默认首屏。

### 2.4 特殊状态

| 状态 | UI |
|---|---|
| 非 Git 仓库 | 空状态：`这里还没有 Git 脉络`，主按钮 `初始化仓库` / `克隆仓库`，如果当前 API 暂不支持 init 则只显示克隆和文档提示。 |
| Clone 进行中 | 使用现有 GitPortal 进度概念，但嵌入 Git Atlas：阶段、速度、当前对象、取消。 |
| Merge conflict | Hero 变 warning/danger，变更地图只突出冲突组；推荐动作改为 `处理冲突`。 |
| Detached HEAD | 显示 `游离 HEAD` 风险解释，主动作 `新建分支保存当前状态`。 |
| 远端未配置 | 同步中心提示添加 remote，提供 URL 输入。 |

### 2.5 动效

- 文件加入提交篮：文件行右侧 badge 从空心到实心，basket count 弹跳 120ms。
- Diff pane 切换：横向 slide 180ms；减少动画时直接切换。
- Commit 成功：timeline 新增点，Hero dirty count 清零，toast 提供 `复制 hash`。
- Pull/Push：箭头沿分支线移动，最多 1.5s；长任务显示真实日志摘要。

## 3. 引导光标 / Guide Beam

### 3.1 用户场景

- Agent 正在执行任务，用户发现理解偏差。
- 当前只能继续输入，消息会排队，纠偏到达太晚。
- 用户需要两种模式：
  - **排队**：默认，安全，等当前工作完成。
  - **引导**：打断当前工作，立即把纠偏发给 agent。

### 3.2 Composer 设计

仅当 `session.thinking === true` 时显示双模式控制：

```text
Agent 正在工作
[ 排队 ] [ 引导 ]
排队：等当前任务完成后发送
引导：打断当前任务并立即纠偏

输入框...
                         [发送]
```

- 默认选中 `排队`。
- 选中 `引导` 后发送按钮文案变为 `引导发送`，颜色使用电光橙。
- 首次使用引导时显示一次轻确认 sheet：
  - 标题：`要打断当前工作吗？`
  - 说明：`Hapi Power 会保留会话记录，并把这条消息优先发送给 Agent。未发送的排队消息不会被删除。`
  - 勾选：`以后不再提示`。
- 后续不反复确认，避免阻碍纠偏。

### 3.3 队列条升级

当前 `QueuedMessagesBar` 只显示 queued。升级为“待发送队列”：

- `排队 2`：普通 queued。
- `引导中`：guide message 已提交，等待 agent 中断确认。
- 如果已有 queued messages，用户发送 guide 后，guide 显示在最前，普通 queued 保持原顺序。
- 每条消息保留编辑/取消，但 guide 一旦 agent 已消费，不可取消，只可在会话中继续纠偏。

### 3.4 会话时间线反馈

在消息流中插入轻量 system event：

- `已选择引导：正在打断当前任务…`
- `Agent 已收到引导消息`
- 失败时：`引导未能立即送达，已改为排队发送`

事件不应占用大量视觉空间，可用 11px text + icon + border-left。

### 3.5 不影响核心会话的 UX 保护

1. 默认永远是排队。
2. Guide 控件只在 thinking 时出现；非 thinking 不制造复杂度。
3. Guide 不删除普通 queued messages。
4. Guide 失败自动降级为 queued，并让用户看见。
5. Abort/Stop 按钮与 Guide 区分：Stop 是停止，Guide 是纠偏并继续。
6. 附件/定时消息默认不允许 guide；如果未来支持，需单独设计上传与中断时序。

## 4. 上下文脉冲 / Context Pulse

### 4.1 新显示规则

用户要求的最小文案：

- 中文：`上下文：40%`
- 英文：`Context: 40%`

颜色：

| 已用比例 | 颜色 | 语义 |
|---|---|---|
| `< 60%` | success green | 健康 |
| `60–80%` | warning yellow | 注意 |
| `> 80%` | danger red | 高风险，建议 compact / 新会话 |

边界建议：`<60` green，`>=60 && <=80` yellow，`>80` red。

### 4.2 组件形态

StatusBar 中替代旧 ctx：

```text
● online    上下文：40%    cache 12k
```

- 小屏只显示 `上下文：40%`，不显示 used/max。
- 桌面 hover/click 弹出 popover：

```text
上下文脉冲
已用：78,400 / 190,000 tokens
占用：41%
缓存命中：12,000
模型：glm-5.1
数据来源：Claude usage + Hapi Power context window
更新：刚刚
```

### 4.3 不可用状态

不能静默消失。显示：

- `上下文：--` 灰色。
- Popover 解释原因：
  - `等待模型返回用量数据`
  - `该供应商未返回 usage 字段`
  - `缺少 context window，无法计算百分比`
  - `当前只收到子任务用量，父会话用量暂不可用`
- 提供操作：`查看诊断`，跳到 API 星桥的 Provider 诊断。

### 4.4 风险提示

- 60–80%：popover 提示 `上下文开始变长，建议在关键节点导出或总结。`
- >80%：提示 `高上下文占用可能增加遗漏、混淆和幻觉风险。建议 /compact 或开启新会话。`
- 不自动弹窗打扰，只在数字变红时轻微 pulse 两次。

### 4.5 动效

- 数字变化：150ms 淡入或轻数字滚动。
- 阈值跨越：颜色过渡 200ms；超过 80% 时红点 pulse 2 次后停止。
- 减少动画：无 pulse，只变色。

## 5. 会话织锦 / Session Loom

### 5.1 产品定位

把现有“大纲”从导航工具升级为 **会话资产工作台**。用户可以：

1. 跳转到关键用户消息。
2. 导出完整但降噪的对话记录。
3. 生成项目设计方案 / 决策日志 / 经验复盘。
4. 下载 Markdown。
5. 对错误引导和理解偏差做检查。

### 5.2 Panel 结构

桌面右侧 panel 宽度从 24rem 升级到响应式 28–36rem；移动端为全屏 drawer。

```text
会话织锦 Session Loom
当前会话标题
[大纲] [导出] [提炼] [资产]

大纲 Tab:
- 用户需求 09:12
- 修改方向 09:16
- 确认边界 09:28

导出 Tab:
范围：完整会话 / 仅用户对话 / 最近 N 条
降噪：过滤“继续/确认”等短回复 ✓
格式：Markdown
[生成预览] [下载 MD]

提炼 Tab:
模板：设计方案 / PRD / 决策日志 / 偏差检查 / 经验卡
深度：快速 / 标准 / 深度
[开始提炼]

资产 Tab:
已生成文档列表
- 2026-06-08-design.md [下载] [复制]
```

### 5.3 导出内容设计

导出的 Markdown 应包括：

```markdown
# 会话导出：{title}

- 会话 ID：...
- 项目路径：...
- Agent：Claude / Codex / ...
- 模型：...
- 导出时间：2026-06-08 14:30
- 时间范围：...
- 过滤规则：过滤短确认、重复继续、纯状态确认

## 1. 会话概要

自动生成 5–10 条摘要。

## 2. 关键需求与约束

- ...

## 3. 原始对话记录

### 09:12 用户
...

### 09:14 AI
...

## 4. 需求澄清问答

| 时间 | 提问方 | 问题 | 回答 | 结论 |

## 5. 可能的偏差与纠正

- ...
```

降噪规则第一版：

- 过滤纯短确认：`继续`、`确认`、`好的`、`可以`、`yes`、`ok`、`go on`。
- 过滤重复空白、系统内部状态、无文本 tool noise。
- 保留需求澄清中的短答，如果它回答了上一条 AI 问题。
- 保留用户明确许可/拒绝，例如“不要实施”“只设计”。

### 5.4 提炼模板

| 模板 | 目标 |
|---|---|
| 设计方案 | 输出项目背景、目标、信息架构、交互、API、验收标准。 |
| PRD 草案 | 输出用户故事、范围、非范围、需求 ID、验收标准。 |
| 决策日志 | 输出关键决策、原因、影响、待确认项。 |
| 偏差检查 | 检查是否出现错误引导、误解、范围漂移、遗漏。 |
| 经验卡 | 抽取可复用 prompt、流程、踩坑、检查清单。 |
| README 片段 | 将会话沉淀为产品文档宣传语。 |

### 5.5 创意扩展

1. **证据地图**：提炼文档每个结论都可展开查看来自哪几条对话。
2. **漂移检测**：比较早期目标与后期执行，标出“范围扩大/方向改变/用户纠偏”。
3. **决策时间线**：把“用户要求/AI 建议/最终采用”串成时间线。
4. **复用片段库**：从会话中提取高质量提示词、验收清单、架构原则。
5. **项目记忆卡**：一键生成可复制到 `.planning/` 的 Markdown seed。
6. **隐私清洗**：导出前扫描 token、密钥、URL、邮箱，提示用户是否遮蔽。
7. **双语导出**：界面 i18n 与文档模板语言分离；用户可选择中文或英文输出。

### 5.6 移动端与 PWA

- 大纲入口在 More 菜单保留，同时在聊天底部可通过小浮标打开。
- Panel 在 iPhone 上全屏，顶部支持下拉关闭。
- 导出生成期间可离开 Panel，完成后显示 toast 和系统分享按钮（Web Share API 可用时）。
- 下载 MD 后提示“已保存到下载”，PWA standalone 下提供复制全文备用。
- 预览使用可折叠章节，避免长文滚动压迫。

### 5.7 动效

- Panel 打开：桌面右侧 slide 180ms；移动端 bottom/fullscreen sheet 上滑 200ms。
- 切换 tab：underline 120ms；内容 fade/translate 80ms。
- 生成提炼：文档骨架逐段出现；减少动画时直接显示进度条。
- 下载完成：按钮从 `下载 MD` 变 `已下载` 1.2s 后恢复。
