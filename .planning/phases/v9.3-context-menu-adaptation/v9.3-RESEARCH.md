## Summary

This phase creates a thin `useContextMenu` wrapper hook around the existing `useLongPress` hook, then applies it to GitStatusPanel file rows. The existing codebase has all needed building blocks: `useLongPress` handles right-click + long-press, `ContextMenu` renders positioned menus, `files.tsx` provides reference patterns for navigation and clipboard, and the `api` client has `readSessionFile` and `getGitDiffFile` endpoints ready.

**Primary recommendation:** Build `useContextMenu.ts` as a 15-line wrapper around `useLongPress`. Add a `MoreIcon` button and context menu to each file row in `GitStatusPanel`. Wire navigation and clipboard callbacks through new props, managed by the parent `git.tsx`.

## Architecture Patterns

### Pattern 1: useContextMenu wrapper hook

**What:** A thin wrapper around `useLongPress` that simplifies the API for context-menu-only use cases.

**When to use:** Any component that needs right-click + long-press to open a context menu.

**Example:**
```typescript
// web/src/hooks/useContextMenu.ts
import { useLongPress } from './useLongPress'

export function useContextMenu(
  onContextMenu: (pos: { x: number; y: number }) => void
) {
  return useLongPress({
    onLongPress: onContextMenu,
    onClick: undefined,  // no click action, only context menu
  })
}
```

The returned value is the full `UseLongPressHandlers` object with 8 handlers. Consumers spread it onto their element:
```tsx
const handlers = useContextMenu((pos) => setMenuPos(pos))
<div {...handlers}>...</div>
```

### Pattern 2: "..." button positioning (from DirectoryTree)

**What:** A button that calculates its position via ref and calls `onContextMenu`.

**From `DirectoryTree.tsx` FileNode (lines 87-102):**
```tsx
const moreRef = useRef<HTMLButtonElement>(null)
// ...
<button
    ref={moreRef}
    type="button"
    className="shrink-0 flex items-center justify-center w-8 h-8 rounded-full text-[var(--app-hint)] hover:bg-[var(--app-subtle-bg)] hover:text-[var(--app-fg)] transition-colors"
    onClick={(e) => {
        e.stopPropagation()
        const rect = moreRef.current?.getBoundingClientRect()
        if (rect) {
            onContextMenu(filePath, 'file', { x: rect.left, y: rect.bottom + 4 })
        }
    }}
>
    <MoreIcon />
</button>
```

### Pattern 3: Context menu state management (from files.tsx)

**From `files.tsx` (lines 237-244, 368-377, 636-648):**
```tsx
// State
const [contextMenu, setContextMenu] = useState<{
    x: number; y: number; path: string; type: 'file' | 'directory'
} | null>(null)
const justClosedRef = useRef(false)

// Handler
const handleContextMenu = useCallback((path, type, point) => {
    if (justClosedRef.current) { justClosedRef.current = false; return }
    setContextMenu((prev) => {
        if (prev && prev.path === path && prev.type === type) return null
        return { ...point, path, type }
    })
}, [])

// Render
{contextMenu && (
    <ContextMenu
        x={contextMenu.x} y={contextMenu.y}
        items={contextMenuItems}
        onClose={() => {
            setContextMenu(null)
            justClosedRef.current = true
            setTimeout(() => { justClosedRef.current = false }, 300)
        }}
    />
)}
```

The `justClosedRef` pattern prevents immediate re-opening when a touch-dismiss triggers the long-press handler on the same position.

### Pattern 4: File navigation via encodeBase64 (from files.tsx)

**From `files.tsx` `handleOpenFile` (lines 272-285):**
```tsx
const handleOpenFile = useCallback((path: string, staged?: boolean) => {
    navigate({
        to: '/sessions/$sessionId/file',
        params: { sessionId },
        search: { path: encodeBase64(path), staged }
    })
}, [navigate, sessionId])
```

The file route (`/sessions/$sessionId/file`) expects:
- `search.path`: base64-encoded file path
- `search.staged`: optional boolean for diff view mode

When `staged` is present, the FilePage shows diff view. When absent, it shows the file content.

### Anti-Patterns to Avoid
- **Do NOT use `onClick` in `useContextMenu`** -- the wrapper is for context-menu-only use, clicking a row should do nothing or navigate
- **Do NOT inline the context menu state in GitStatusPanel** -- let `git.tsx` manage it so callbacks have access to `api`, `navigate`, and `sessionId`
- **Do NOT duplicate the MoreIcon** -- import it from DirectoryTree or extract to shared icons file

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Right-click + long-press detection | Custom event handlers | `useLongPress` | Already handles touch/mouse coordination, iOS text selection clearing, pointer move threshold |
| Clipboard copy | `navigator.clipboard.writeText` directly | `useCopyToClipboard` hook | Falls back to `execCommand`, adds haptic feedback |
| Menu positioning | Manual viewport clamping | `ContextMenu` component | Already handles safe-area-inset, viewport bounds, click-outside dismissal, ESC key |
| File path encoding for URLs | `btoa(path)` | `encodeBase64` from `@/lib/utils` | Handles UTF-8 correctly via TextEncoder |

## Common Pitfalls

### Pitfall 1: "..." button click propagates to row
**What goes wrong:** Clicking "..." also triggers the row's onClick handler, causing unintended navigation.
**How to avoid:** Always call `e.stopPropagation()` in the "..." button's onClick handler. DirectoryTree already does this correctly.
**Warning signs:** Menu opens and page navigates simultaneously.

### Pitfall 2: Context menu re-opens immediately on touch dismiss
**What goes wrong:** On mobile, dismissing a context menu via touch triggers the long-press handler again.
**How to avoid:** Use the `justClosedRef` pattern from `files.tsx` -- set to `true` on close, reset after 300ms, check before opening.
**Warning signs:** Context menu flickers open/closed rapidly on mobile.

### Pitfall 3: Missing `staged` flag in diff navigation
**What goes wrong:** Navigating to diff view for an unstaged file passes `staged: true`, showing wrong diff.
**How to avoid:** GitStatusPanel's `GitFile` type has a `status` string like "M", "A", "?", but no `staged` boolean. The `parseGitStatus` function in GitStatusPanel does not distinguish staged vs unstaged files. For v9.3, pass `staged` as `undefined` (the file page will show the working tree diff). If distinction is needed later, the parser must be enhanced.
**Warning signs:** Diff shows staged changes when user expected unstaged changes.

### Pitfall 4: i18n key mismatch
**What goes wrong:** Adding keys under `git.context.*` but the existing pattern uses flat dot-notation strings in a single object.
**How to avoid:** Follow the exact pattern: `'git.context.viewDiff': 'View Diff'` in the locale objects. The `t()` function does a simple object lookup.
**Warning signs:** Menu items show raw keys like `git.context.viewDiff` instead of translated text.

**Validation needed:** A1 -- the `parseGitStatus` function extracts `xy[0]` or `xy[1]` as single status char but does not track which side of XY it came from. For v9.3 this is acceptable since all 3 actions (view diff, copy path, open file) work with the file path alone.

## Open Questions

1. **Should "Open File" navigate to file viewer or file editor?**
   - What we know: The file route at `/sessions/$sessionId/file` shows file content with optional diff. There is no separate "editor" route.
   - Recommendation: Navigate to file viewer without `staged` param. The file page has edit mode toggle.

2. **Should the "..." button icon be extracted to a shared component?**
   - What we know: DirectoryTree has its own inline `MoreIcon` component (3 SVG circles). Same icon would be duplicated.
   - Recommendation: For v9.3, duplicate is fine (it's 5 lines). Extract to shared icons in a future cleanup.

3. **How to handle "View Diff" when the file is untracked (status "?")?**
   - What we know: Untracked files have no diff content. `getGitDiffFile` would return empty or error for untracked files.
   - Recommendation: Disable "View Diff" menu item for untracked files (`disabled: status === '?'`).
