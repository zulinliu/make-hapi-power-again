# Phase v9.4: Git 状态文件预览 - Research

**Researched:** 2026-06-03
**Domain:** File preview UI with syntax highlighting in Git status panel
**Confidence:** HIGH

## Summary

This phase adds a file preview capability to the Git Status tab. Clicking a changed file opens a preview panel (slide-over on desktop, bottom sheet on mobile) that displays the file content with Shiki syntax highlighting. The implementation leverages several existing components and patterns already established in the codebase: `readSessionFile` API (returns base64-encoded content), `CodeBlock` (Shiki rendering), `ImagePreview` (image files), `decodeBase64`/`isBinaryContent` utilities, and the `Dialog` pattern from Radix UI.

**Primary recommendation:** Build a single `GitFilePreview` component that handles all file types (text, image, markdown, binary, deleted). Wire it into `GitFileRow` via a new `onPreview` callback. Use CSS transform transitions for the slide-over animation with a `md:` breakpoint to switch between desktop side-panel and mobile full-screen sheet modes. No new dependencies needed -- everything is already in the codebase.

```typescript
// Source: VERIFIED from web/src/lib/shiki.ts:105-111
function resolveLanguage(lang: string | undefined): string {
    if (!lang) return 'text'
    const cleaned = lang.startsWith('language-') ? lang.slice('language-'.length) : lang
    const lower = cleaned.toLowerCase().trim()
    if (lower === 'text' || lower === 'plaintext' || lower === 'txt') return 'text'
    return langAlias[lower] ?? lower
}

// Usage for file path -> language:
// 1. Get extension: const ext = path.split('.').pop()?.toLowerCase() ?? ''
// 2. Map through langAlias: langAlias[ext] ?? ext
// This handles sh->shellscript, js->javascript, py->python, etc.
```

### Anti-Patterns to Avoid

- **Don't use Dialog component for the preview panel:** The existing Radix Dialog is a centered modal. The slide-over needs custom positioning (right edge / bottom). Build a custom overlay using the same Portal + Overlay pattern but with different positioning.
- **Don't re-implement binary detection or base64 decoding:** These are already in the codebase at `isBinaryContent` (file.tsx:49) and `decodeBase64` (utils.ts:11). Copy the utility functions or extract them to a shared location.
- **Don't use useContextMenu's onClick for preview:** `useContextMenu` is configured with `onClick: undefined` in `useLongPress`. The preview trigger must be a separate click handler on the GitFileRow div.

## Common Pitfalls

### Pitfall 1: useContextMenu Click Conflict
**What goes wrong:** GitFileRow uses `useContextMenu` which spreads handlers including onMouseDown/onMouseUp/onTouchStart/onTouchEnd. Adding a direct onClick would conflict with the long-press detection.
**Why it happens:** `useLongPress` (which `useContextMenu` wraps) uses onClick: undefined, meaning regular clicks do nothing. But the mouseDown/mouseUp handlers are still active.
**How to avoid:** Add the preview click as a separate handler. Since useContextMenu has `onClick: undefined`, the handleEnd in useLongPress won't fire a click. Add an `onClick` handler directly on the GitFileRow div that calls `onPreview`. The spread `{...handlers}` will provide right-click/long-press support, and the explicit onClick will handle the preview trigger.
**Warning signs:** Long press triggers preview instead of context menu, or right-click opens preview.

### Pitfall 2: Binary Content Decode Errors
**What goes wrong:** Large binary files (images, executables) will fail base64 decoding or produce garbled text.
**Why it happens:** The API returns base64 for all files. `decodeBase64` uses `atob` which works on the base64 string, then `TextDecoder('utf-8')` which may produce replacement characters for non-UTF-8 data.
**How to avoid:** Always check `decodedContentResult.ok` AND `isBinaryContent()` before rendering as text. For images, use the raw base64 content directly with `data:` URI without decoding. Check `resolveImageMimeType` BEFORE attempting text decode for better UX.
**Warning signs:** Garbled text display, console warnings about invalid UTF-8.

### Pitfall 3: Deleted File Content Not Available
**What goes wrong:** Calling readSessionFile on a deleted file (status === 'D') may fail or return old content from the working tree.
**Why it happens:** The API reads from the current working tree. Deleted files no longer exist there.
**How to avoid:** Per D-04, check status === 'D' first and show "File deleted" message without making the API call (set `enabled: false` on the query).
**Warning signs:** API error or unexpected content for deleted files.

### Pitfall 4: Large File Performance
**What goes wrong:** Very large files (e.g., minified JS, large JSON) cause slow rendering and high memory usage in CodeBlock/Shiki.
**Why it happens:** Shiki parses and highlights every line. CodeBlock already has collapse logic (>18 lines or >1800 chars), but the initial highlight still processes the full content.
**How to avoid:** CodeBlock's `collapseLongContent` prop handles visual truncation. Consider adding a size check before highlighting -- if content exceeds a threshold (e.g., 500KB), show a message instead of attempting to highlight.
**Warning signs:** Browser freeze when previewing large files.

### Pitfall 5: Mobile Sheet Swipe-to-Close
**What goes wrong:** The bottom sheet on mobile needs a drag-down gesture to dismiss, but this can conflict with scroll inside the preview.
**Why it happens:** Both gesture and scroll use touch events on the same axis (vertical).
**How to avoid:** Only enable swipe-to-close on the drag handle area at the top of the sheet, not on the content area. The content area should scroll normally.
**Warning signs:** Cannot scroll preview content on mobile, or sheet closes when trying to scroll.

### Pitfall 6: Query Key Collision
**What goes wrong:** Using `queryKeys.sessionFile(sessionId, path)` for the preview may share cache with the file.tsx page, which could be stale after git changes.
**Why it happens:** React Query caches by key. If the user previews a file, navigates away, and the file changes, the cached data is stale.
**How to avoid:** This is actually fine -- the shared cache means if the user already viewed a file on the file page, the preview loads instantly. The cache invalidation strategy (staleTime, refetchOnWindowFocus) handles freshness. No special handling needed.
**Warning signs:** None expected.

### Base64 Decoding

```typescript
// Source: VERIFIED from web/src/lib/utils.ts:11-20
export function decodeBase64(value: string): { text: string; ok: boolean } {
    try {
        const binaryString = atob(value)
        const bytes = Uint8Array.from(binaryString, (char) => char.charCodeAt(0))
        const text = new TextDecoder('utf-8').decode(bytes)
        return { text, ok: true }
    } catch {
        return { text: '', ok: false }
    }
}
```

### Binary Content Detection

```typescript
// Source: VERIFIED from web/src/routes/sessions/file.tsx:49-57
function isBinaryContent(content: string): boolean {
    if (!content) return false
    if (content.includes('\0')) return true
    const nonPrintable = content.split('').filter((char) => {
        const code = char.charCodeAt(0)
        return code < 32 && code !== 9 && code !== 10 && code !== 13
    }).length
    return nonPrintable / content.length > 0.1
}
```

### Image MIME Type Resolution

```typescript
// Source: VERIFIED from web/src/routes/sessions/file.tsx:22-27, 39-42
const IMAGE_MIME_BY_EXTENSION: Record<string, string> = {
    apng: 'image/apng', avif: 'image/avif', bmp: 'image/bmp',
    gif: 'image/gif', ico: 'image/x-icon', jpeg: 'image/jpeg',
    jpg: 'image/jpeg', png: 'image/png', svg: 'image/svg+xml',
    tif: 'image/tiff', tiff: 'image/tiff', webp: 'image/webp'
}

function resolveImageMimeType(path: string): string | null {
    const ext = path.split('.').pop()?.toLowerCase()
    if (!ext) return null
    return IMAGE_MIME_BY_EXTENSION[ext] ?? null
}
```
