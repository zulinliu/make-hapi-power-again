## 1. PTY 终端技术验证

### 关键发现：hapi 不使用 node-pty

hapi 的终端实现完全依赖 Bun 的原生 API，**不使用 node-pty**：

- **PTY 创建**：`bun.spawn(shellCommand, { terminal: { cols, rows, data, exit } })`
  - 文件：`cli/src/terminal/TerminalManager.ts:180`
  - `data` 回调接收 `Uint8Array` 输出，通过 `TextDecoder` 解码
  - Shell 检测：`$SHELL` → `/bin/zsh` → `/bin/bash` → PowerShell
  - 环境变量过滤（剥离 `CLI_API_TOKEN`、`ANTHROPIC_API_KEY` 等敏感键）
  - 默认 `TERM=xterm-256color`，`COLORTERM=truecolor`

- **PTY 销毁**（`TerminalManager.ts:298-322`）：
  - `cleanup()` 终止子进程 + `runtime.terminal.close()`
  - 空闲超时（默认 15 分钟，`HAPI_TERMINAL_IDLE_TIMEOUT_MS`）
  - 每会话最大 4 个终端（`HAPI_TERMINAL_MAX_TERMINALS`）

- **PTY Resize**（`TerminalManager.ts:258-267`）：
  - `runtime.terminal.resize(cols, rows)` 直接调用

- **运行时检查**：`if (!bun || typeof bun.spawn !== 'function')` → "Terminal is unavailable in this runtime"

### dinotty 的 PTY 实现（Rust 参考）

- 使用 `portable-pty` 0.8 跨平台 PTY 抽象
- `create_session()` 创建 PTY 对，初始 80x24
- 通过 `tokio::task::spawn_blocking` 处理阻塞 PTY 读取
- `VirtualScreen`（VT220 终端仿真器）支持有状态重连
- CWD 跟踪通过 OSC 0 标题转义序列嗅探
- 后台清理任务每 30 秒移除分离超过 5 分钟的会话

### Phase 0.5 PoC 调整

**原计划**：验证 node-pty + Bun 兼容性
**实际情况**：hapi 使用 Bun 原生 `terminal` API，无需 node-pty

**新 PoC 目标**：
1. 验证 `bun.spawn({ terminal })` 在 Linux 上正常工作
2. 验证终端输出正确流式传输到 xterm.js
3. 验证 resize 事件正确传递
4. 评估是否需要为 v0.2 增加资源限制（rlimit 等需要 Bun 支持）

## 2. 实时通信技术验证

### hapi 传输层

- **Socket.IO v4.8.3** + `@socket.io/bun-engine`（Bun 专用引擎）
- **三个命名空间**：
  1. `/cli` — CLI 连接认证（共享密钥 token）
  2. `/terminal` — Web 终端连接认证（JWT，含 uid + namespace）
  3. 默认 `/` — 未使用

- **数据流路径**：
  ```
  Web Client → Socket.IO /terminal → Hub → /cli namespace → CLI TerminalManager
  CLI TerminalManager.onOutput → /cli terminal:output → Hub → Web Client /terminal
  ```

- **二进制帧**：终端数据作为 **字符串** 传输，非二进制。`TerminalOutputPayloadSchema` 定义 `data: z.string()`

- **SSE 用于状态同步**：会话列表、消息、机器状态通过 SSE（`/api/events`）推送，非 Socket.IO

### dinotty 传输层（Rust 参考）

- 原生 WebSocket（axum ws）
- 6 个 WebSocket 通道：终端、同步、文件监视、系统监控、通知、历史
- 双传输抽象：WebSocket + Tauri IPC（`Transport` 接口封装两种模式）
- 会话多客户端：`DashMap` + `mpsc::unbounded_channel` 广播

### Phase 0.5 PoC 目标

1. 验证 Socket.IO binary event 传输图片数据（设计文档 ADR-001 要求）
2. 验证 `@socket.io/bun-engine` 支持二进制事件
3. 评估是否需要切换到字符串 base64 编码（备选方案）

## 3. Git 操作技术验证

### hapi 现有 Git 能力

hapi 已有基础 Git 功能（通过 RPC 调用）：

- **RPC 方法**（`shared/src/rpcMethods.ts`）：
  - `GitStatus` → `git status --porcelain=v2 --branch --untracked-files=all`
  - `GitDiffNumstat` → `git diff --numstat`
  - `GitDiffFile` → `git diff --no-ext-diff -- <file>`

- **Hub REST 路由**（`hub/src/web/routes/git.ts`）：
  - `GET /api/sessions/:id/git-status`
  - `GET /api/sessions/:id/git-diff-numstat`
  - `GET /api/sessions/:id/git-diff-file`

- **CLI 端**：通过 `child_process.execFile('git', ...)` 执行，10s 超时
- **路径安全**：`validatePath(targetPath, workingDirectory)` 防路径遍历

### dinotty 的 Git 能力（Rust 参考）

- 文件工作区 API 包含 git-status、git-diff、git-stage-lines、git-revert-lines
- 通过 `git2` crate（libgit2 绑定）实现

### Phase 0.5 PoC 目标

1. 验证 hapi 现有 Git 功能在 Linux 上正常工作
2. 评估 isomorphic-git 在浏览器端执行 clone/status/diff 的可行性
3. 决定 Git 操作是走 CLI RPC 还是 Hub 直接调用

## 5. 插件系统技术验证

### hapi 现状

**hapi 没有插件系统**。扩展能力通过静态注册 RPC handler 实现：
- `registerCommonHandlers()` 在启动时注册所有 RPC 处理器
- 新能力通过创建新 handler 文件 + 注册调用添加

### dinotty 的插件系统（Rust 参考）

- CSS 作为 `<style>` 元素注入
- 插件 API：terminal、storage、commands、ui、settings
- 热重载：文件监视 + 500ms 防抖 + WebSocket 同步通道广播

## 6. 安全验证

### hapi 现有安全措施

- **路径安全**：`validatePath(targetPath, workingDirectory)` — resolve + startsWith 检查
- **命名空间隔离**：所有查询包含 namespace 过滤
- **JWT 认证**：终端 Socket.IO 使用 JWT（含 uid + namespace）
- **共享密钥认证**：CLI Socket.IO 使用共享 token

### dinotty 的安全措施（Rust 参考）

- Token 认证 + IP 白名单（CIDR + 通配符）
- 暴力破解保护（60s 内 5 次失败锁定）
- 常数时间 token 比较
- 插件路径遍历保护

## 8. 架构洞察

### 可复用模式（来自 dinotty）

1. **DashMap + 广播通道** — 并发安全会话管理 + 多客户端广播
2. **VirtualScreen** — VT220 终端仿真器，支持有状态重连
3. **双传输抽象** — 单一 Transport 接口封装 WebSocket + IPC
4. **插件热重载** — 文件监视 + blob URL 动态加载 + WebSocket 同步
5. **OSC 标题 CWD 跟踪** — 从 shell 标题推断工作目录

### 关键架构决策

基于研究结果，Phase 0.5 需确认以下决策：

1. **终端方案**：沿用 Bun `terminal` API（非 node-pty），在 CLI 层管理 PTY
2. **二进制传输**：验证 Socket.IO binary event 可行性，不可行则用 base64 字符串
3. **Git 方案**：基础操作走 CLI RPC（已有），高级操作（clone/branch/merge）通过 Hub 直连 isomorphic-git
4. **插件方案**：Blob URL 动态加载 + ErrorBoundary 隔离（参考 dinotty 模式）
5. **文件方案**：基础操作走 CLI RPC（已有），扩展操作（上传/下载/zip）在 Hub 层实现
