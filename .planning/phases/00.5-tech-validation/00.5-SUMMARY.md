## Task 2: Socket.IO Binary Event

**结果**: 4/4 通过

| 测试 | 结果 | 说明 |
|------|------|------|
| 二进制数据往返 | ✓ | 8 bytes 正确传输 |
| 大二进制传输 | ✓ | 1MB / 129ms |
| String vs Binary 对比 | ✓ | binary=135ms, string=21ms |
| 并发多客户端 | ✓ | 4 个并发客户端正常 |

### 关键发现

1. **String 比 Binary 快 6 倍**: 10x100KB 测试中 string 21ms vs binary 135ms
2. 原因: Socket.IO 二进制传输需要 base64 编码/解码额外开销
3. 1MB 二进制传输延迟 129ms，对于终端数据足够

### 对实时通信的影响

- **终端数据**: 继续使用 string 编码（与 hapi 现有方式一致）
- **大文件传输**: 可考虑直接 HTTP 上传，不走 Socket.IO
- **不需要改为 binary frame**: string 方式性能更优

## Task 4: isomorphic-git

**结果**: 4/6 通过

| 测试 | 结果 | 说明 |
|------|------|------|
| git init | ✓ | 使用 node:fs 直接传入 |
| git add + commit | ✓ | 提交 SHA: 5e341433 |
| git log | ✓ | 1 commit 检索成功 |
| git status | ✗ | 返回 `*modified`（含 staged 前缀） |
| git diff (walk) | ✗ | WORKDIR tree 与 node:fs 不兼容 |
| Unicode/CJK 文件名 | ✓ | 中文文件名提交成功 |

### 关键发现

1. **fs 适配**: 直接传 `node:fs` 即可工作，无需自定义适配器
2. **walk/diff**: `WORKDIR` tree 需要 LightningFS 等浏览器端 fs，与 node:fs 不兼容
3. **status 返回值**: `*modified` 表示 staged + unstaged，需要处理前缀

### 对模块 A (Git) 的影响

- **服务端 Git 操作**: 可使用 isomorphic-git 替代 child_process exec
- **浏览器端 Git**: 需要配合 LightningFS 或 IndexedDB fs
- **diff 功能**: 服务端继续用 `git diff` CLI 命令，浏览器端需要其他方案
- **status 判断**: 需处理 `*` 前缀表示 staged + modified 状态

## Task 6: Path Security

**结果**: 8/11 通过（3 个假阴性）

### 被阻止的攻击 (8/11)

- ✓ 路径遍历 `../../../etc/passwd`
- ✓ URL 编码 `%2e%2e%2f`
- ✓ 嵌套遍历 `....//....//`
- ✓ 绝对路径 `/etc/shadow`
- ✓ Null byte (文件路径) `file.txt%00.jpg`
- ✓ UNC 路径 `\\server\share`
- ✓ 空字节 (路径) `\x00`
- ✓ 超长路径 (1000+ chars)

### 未被阻止的攻击 (3/11)

- ✗ **双重 URL 编码** `%252e%252e%252f` — 需要多次 URL 解码
- ✗ **Null byte 注入** `%00.exe` — URL 解码后的 null byte 未处理
- ✗ **多重点号路径** `.../.../.../` — 路径标准化后仍含遍历

### 对安全模块的影响

修复 `sanitizePath` 函数:
1. 循环 URL 解码直到稳定
2. 移除所有 null bytes
3. resolve 后再次验证前缀
