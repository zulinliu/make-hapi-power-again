## F.1 消息排队系统

### F.1.1 问题场景

AI 代理处理用户请求时处于忙碌状态，开发者可能有新的想法或补充信息需要传达：
- 代理正在重构认证模块，开发者突然想到还需要处理 token 刷新逻辑
- 代理执行长任务时，开发者想补充上下文或调整方向
- 代理卡在某个问题上，开发者想直接干预引导方向

### F.1.2 核心设计决策

1. **双模式设计** — 默认排队模式（消息入队等待），可选引导模式（立即发送/中断）
2. **客户端队列** — 消息队列存储在前端状态中，代理完成当前轮次时自动出队
3. **FIFO + 可编辑** — 先入先出，但支持编辑、重排、立即发送、删除
4. **无缝切换** — 两种模式间一键切换，切换不影响已排队的消息

### F.1.3 数据模型

```typescript
// 排队消息
interface QueuedMessage {
  id: string                       // UUID
  sessionId: string
  agentId: string
  text: string
  attachments: MessageAttachment[]
  queuedAt: number                 // 入队时间戳

// 消息附件

  kind: 'image' | 'file'
  name: string
  size: number
  mimeType: string
  data?: string                    // base64 (图片) 或路径 (文件)

// 发送模式
type MessageMode = 'queue' | 'guide'

// 队列状态（前端 Zustand 存储）
interface MessageQueueState {
  mode: MessageMode                // 当前发送模式
  queues: Map<string, {            // agentId → 消息队列
    messages: QueuedMessage[]
  }>
}

// 队列操作结果
type QueueOperationResult =
  | { type: 'queued'; message: QueuedMessage }
  | { type: 'submitted'; messageId: string }
  | { type: 'failed'; error: string }
  | { type: 'noop' }
```

本功能主要为客户端行为，服务端仅提供消息发送 API：

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/sessions/:id/agents/:agentId/messages` | 发送消息（排队或直发） |
| GET | `/sessions/:id/agents/:agentId/status` | 查询代理状态（busy/idle） |

### F.1.5 核心流程

```
用户点击发送
     │
     ├── mode === 'queue' && agentStatus === 'running'?
     │       │
     │       ├── 是 → 消息入队，显示在队列轨道
     │       │         返回 { type: 'queued' }
     │       │
     │       └── 否 → 直接发送
     │                 返回 { type: 'submitted' }
     │
     └── mode === 'guide'?
             │
             ├── 是 → 立即发送（附带中断标记）
             │         代理收到 guide 标记，可中断当前操作
             │         返回 { type: 'submitted' }
             │
             └── 否 → 走默认排队逻辑
```

### F.1.6 自动出队流程

```
代理完成当前轮次（状态从 running → idle）
     │
     ├── 检查该 agentId 的消息队列
     │       │
     │       ├── 队列非空 → 取出第一条消息
     │       │              发送到代理
     │       │              更新队列（移除已发送消息）
     │       │
     │       └── 队列为空 → 无操作
     │
     └── 如果有多条排队消息
              → 只发送第一条
              → 剩余消息等下一轮次依次发送
```

### F.1.8 交互设计

1. **模式切换** — 输入框右侧的模式切换按钮，点击在排队/引导间切换，切换时有 tooltip 提示差异
2. **入队反馈** — 消息入队时，输入框上方出现队列轨道，显示排队消息，附带微动画（从下向上滑入）
3. **编辑排队消息** — 点击编辑按钮，消息从队列移除，文本回填到输入框，用户可修改后重新发送
4. **立即发送** — 点击立即发送按钮，消息从队列移除并直接发送，即使代理忙碌
5. **代理空闲通知** — 代理从忙碌变为空闲时，如果队列中有消息，短暂显示"正在发送排队消息..."提示
6. **队列上限** — 单代理最多排队 10 条消息，超出时输入框禁用并提示"队列已满，请等待代理处理"

### F.1.9 引导模式设计

引导模式适用于开发者需要直接干预代理方向的场景：
- 发送的消息携带 `guide: true` 标记
- 代理收到引导消息后，可中断当前操作，优先处理引导指令
- 引导模式发送的消息**不进入队列**，即使代理忙碌也立即送达
- 切换到引导模式时显示一次性提示："引导模式：消息将立即发送，可中断代理当前操作"

### F.1.10 与代理状态的联动

```typescript
// 前端监听代理状态变化
function onAgentStatusChange(agentId: string, prevStatus: string, newStatus: string) {
  if (prevStatus === 'running' && newStatus !== 'running') {
    // 代理完成一轮操作，检查队列
    const queue = messageQueueStore.getQueue(agentId)
    if (queue.length > 0) {
      const [next, ...rest] = queue
      sendMessage(agentId, next.text, next.attachments)
      messageQueueStore.setQueue(agentId, rest)
    }
  }
}
```

## F.2 语音助手

### F.2.1 问题场景

- 开发者双手在键盘上编码，想口头给代理下达指令
- 在移动设备上，语音输入比文字输入更高效
- 代理完成任务后需要语音通知开发者
- 开发者想听到代理的回复而非阅读长文本

### F.2.2 核心设计决策

1. **四槽位语音提供者架构** — 参考 Paseo 的独立 provider slot 设计，每个功能可独立选择云端或本地
2. **离线优先** — 本地 Sherpa-ONNX 作为默认提供者，不依赖外部 API
3. **双模式语音** — 语音对话模式（全双工 VAD + STT + TTS）+ 听写模式（流式 STT → 文本输入）
4. **动作能力** — 参考 Happier 的语音代理思路，语音指令可触发代理动作（不只是对话）

## F.3 二进制帧复用

### F.3.1 问题场景

WebSocket 通信中，终端输出、文件传输等二进制数据与 JSON 控制消息混合传输时存在效率问题：
- 终端输出频繁产生小块二进制数据，JSON 封装开销大
- 大文件传输如果通过 base64 编码，体积膨胀 33%
- 多个终端实例的输出需要区分，缺少多路复用机制

### F.3.2 核心设计决策

1. **基于 Socket.IO binary event 的多路复用** — 二进制帧通过 Socket.IO binary emit 传输，与 JSON 消息共享同一连接（ADR-001）
2. **双域设计** — 终端流和文件传输使用不同操作码范围，互不冲突
3. **零拷贝设计** — 二进制数据直接以 Buffer/ArrayBuffer 传输，不经过 JSON 序列化
4. **向后兼容** — 现有 JSON 消息协议不变，二进制帧作为扩展能力渐进引入

### F.3.3 帧格式设计

#### 终端流帧

```
操作码范围：0x01 - 0x0F

+--------+--------+=========+
| Opcode | Slot   | Payload |
| 1 byte | 1 byte | N bytes |
+--------+--------+---------+

Opcode:
  0x01  Output    (服务端→客户端：终端输出)
  0x02  Input     (客户端→服务端：键盘输入)
  0x03  Resize    (客户端→服务端：终端尺寸变更)
  0x04  Snapshot  (服务端→客户端：完整终端状态快照)
  0x05  Restore   (服务端→客户端：ANSI 渲染恢复，用于迟加入客户端)

Slot: 终端实例标识（0-255），支持多终端多路复用

Payload:
  Output/Input: 原始字节（UTF-8 文本，通常为终端转义序列）
  Resize: JSON 编码 { rows: number, cols: number }
  Snapshot: JSON 编码 TerminalState
  Restore: ANSI 渲染字节
```

#### 文件传输帧

```
操作码范围：0x10 - 0x1F

FileBegin (0x10):
+--------+-----------+----------+----------+===========+
| Opcode | ReqId Len | ReqId    | Meta Len | Metadata  |
| 1 byte | 1 byte    | N bytes  | 2 bytes  | M bytes   |
+--------+-----------+----------+----------+-----------+

FileChunk (0x11):
+--------+-----------+----------+===========+
| Opcode | ReqId Len | ReqId    | Payload   |
| 1 byte | 1 byte    | N bytes  | N bytes   |
+--------+-----------+----------+-----------+

FileEnd (0x12):
+--------+-----------+----------+
| Opcode | ReqId Len | ReqId    |
| 1 byte | 1 byte    | N bytes  |
+--------+-----------+----------+

FileBegin Metadata (JSON):
{
  "mime": string,          // MIME 类型
  "size": number,          // 文件总大小
  "encoding": "utf-8" | "binary",
  "modifiedAt": string,    // ISO 时间戳
  "filename": string       // 文件名
}
```

### F.3.4 多路复用策略

```typescript
// Socket.IO binary event 接收（ADR-001）
socket.on('binary', (data: ArrayBuffer) => {
  const bytes = new Uint8Array(data)

  // 尝试解码为二进制帧
  const frame = tryDecodeBinaryFrame(bytes)
  if (frame) {
    dispatchBinaryFrame(frame)
    return
  }

  // 非二进制帧 → 不应到达此处（JSON 消息走普通 event）
  logger.warn('Unknown binary frame received')
})

// Socket.IO binary event 发送
function sendBinaryFrame(socket: Socket, frame: Buffer): void {
  socket.emit('binary', frame)  // Socket.IO 自动处理 binary
}

// 二进制帧解码
function tryDecodeBinaryFrame(bytes: Uint8Array): BinaryFrame | null {
  if (bytes.length < 2) return null

  const opcode = bytes[0]

  // 终端帧范围
  if (opcode >= 0x01 && opcode <= 0x0F) {
    return decodeTerminalFrame(bytes)
  }

  // 文件传输帧范围
  if (opcode >= 0x10 && opcode <= 0x1F) {
    return decodeFileTransferFrame(bytes)
  }

  return null // 非二进制帧
}
```

### F.3.10 安全考虑

- 二进制帧在 Socket.IO 握手认证通过后才接受，认证前丢弃所有帧
- 文件传输帧大小限制：单块最大 1MB，单文件最大 500MB
- 终端输出帧大小限制：单帧最大 256KB，超出降级为快照模式
- requestId 长度限制：最大 255 字节 UTF-8 编码
- FileBegin 元数据限制：最大 65535 字节
- 文件传输路径通过 sanitizePath() 校验，防止路径遍历

## F.4 多代理编排 Skills

### F.4.1 问题场景

复杂开发任务需要多个 AI 代理协作完成：
- 前端重构需要"实现 → 验证 → 修复"循环
- 复杂 bug 需要"分析 → 方案对比 → 实施"的多角色讨论
- 长任务需要分解为多个阶段，每个阶段由专门代理执行
- 需要第二个专业意见来验证方案可行性

### F.4.2 核心设计决策

1. **Skill 驱动编排** — 编排逻辑以 Skill 形式定义，每个 Skill 描述编排流程、角色分配和退出条件
2. **生命周期状态机** — 代理遵循统一的生命周期（初始化→空闲→运行→空闲/错误→关闭）
3. **父子关系** — 父代理可创建子代理，子代理完成后通知父代理，支持级联归档
4. **通过 Skill 组合保证质量** — 不同 Skill 组合覆盖不同场景的质量需求

### F.4.4 编排 Skill 类型

#### 1. Loop（循环验证）

```
适用场景：实现功能 → 运行测试 → 失败修复 → 重复直到通过

流程：
┌──────────────────────────────────────────┐
│ Worker 代理执行任务                        │
│     │                                     │
│     ├── 运行验证检查（Shell 命令）          │
│     │       │                             │
│     │       ├── 通过 → 运行验证代理确认     │
│     │       │       │                     │
│     │       │       ├── 确认通过 → 成功退出 │
│     │       │       └── 确认失败 → 继续循环 │
│     │       │                             │
│     │       └── 失败 → 继续循环            │
│     │                                     │
│     └── 达到最大迭代次数 → 失败退出         │
└──────────────────────────────────────────┘

应用示例：
- "驱动测试到通过" — 运行测试 → 修复 → 重复
- "PR 看护" — 持续检查 PR 状态并修复 CI 问题
- "代码标准化" — lint → 修复 → 重复
```

#### 2. Handoff（任务交接）

```
适用场景：将任务从一个代理交接给另一个，携带完整上下文

流程：
┌──────────────────────────────────────────┐
│ 源代理总结当前状态                         │
│     │                                     │
│     ├── 上下文（当前任务描述）              │
│     ├── 决策记录（已做出的选择）            │
│     ├── 失败尝试（已排除的方案）            │
│     ├── 约束条件（必须遵守的限制）          │
│     │                                     │
│     └── 创建新代理，传入自包含简报           │
│         新代理从简报开始，无历史负担         │
└──────────────────────────────────────────┘

应用示例：
- "上下文过大时换新代理" — 避免长会话退化
- "跨专业领域交接" — 后端代理完成 API 后交接给前端代理
- "工作树隔离" — 在独立工作树中继续开发
```

#### 3. Advisor（顾问意见）

```
适用场景：获取第二个专业意见，验证方案可行性

流程：
┌──────────────────────────────────────────┐
│ 主代理完成方案/实现                        │
│     │                                     │
│     ├── 创建顾问代理（可选不同提供者/模型）  │
│     │                                     │

│     │                                     │
│     ├── 顾问返回分析结果                    │

│     │   - 风险评估                         │
│     │                                     │
│     └── 主代理根据意见调整方案              │
└──────────────────────────────────────────┘

应用示例：
- "方案审查" — Opus 审查 Sonnet 的方案
- "安全审计" — 专门的安全审查代理
- "性能评估" — 性能优化建议
```

#### 4. Committee（委员会讨论）

```
适用场景：复杂问题需要多角度分析，两个高推理能力代理讨论

流程：
┌──────────────────────────────────────────┐
│ 阶段 1：并行分析                          │
│   ├── 代理 A（如 Claude Opus）分析问题     │
│   └── 代理 B（如 GPT-4）分析问题           │
│     │                                     │

│   └── 编排者或指定代理执行                  │
│     │                                     │

应用示例：
```

#### 5. Epic（长任务编排）

```

│ 1. 收集偏好    → 读取 orchestration-preferences.json      │

└──────────────────────────────────────────────────────────┘

应用示例：
- "大规模重构" — 研究→规划→审查→实施→验证→交付
- "新功能开发" — 研究→设计→实施→测试→交付
```

## F.8 关键权衡

| 决策 | 选择 | 理由 |
|------|------|------|
| 消息队列存储 | 前端 Zustand | 纯客户端行为，无需服务端参与，减少网络往返 |
| 语音离线优先 | Sherpa-ONNX Worker 进程 | 隐私保护 + 无网络依赖 + 无 API 成本 |
| 二进制帧多路复用 | Socket.IO binary event + 首字节操作码 | 复用现有连接，零开销区分，兼容 JSON 协议（ADR-001） |
| 代理生命周期 | 统一状态机 | 所有编排 Skill 共享相同生命周期，降低复杂度 |
| 编排 Skill 格式 | SKILL.md 声明式 | 与现有 Skill 系统一致，用户可自定义 |
| 子代理通知 | 事件订阅 | 父代理无需轮询，子代理完成/出错自动通知 |
| 委员会只读 | 委员会成员不直接编辑 | 避免多代理并发编辑冲突，保持单一作者原则 |
