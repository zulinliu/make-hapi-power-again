## 1. 项目概述

### 1.1 项目定位

**Hapi Power** 是一个面向 AI 编码代理的全栈开发者工作台，将远程代理管理、本地开发工具链、扩展生态系统融合为统一的 Web 体验。

一句话定义：**让 AI 编码代理拥有完整的开发者环境 — 代码编辑、终端操作、版本控制、插件扩展，全部在浏览器中完成。**

### 1.2 解决什么问题

当前 AI 编码代理（如 Claude Code、Cursor 等）的管理界面通常只提供对话和消息流，开发者仍需在本地 IDE 和远程管理界面之间频繁切换。这带来了三个核心痛点：

1. **上下文断裂** — AI 代理修改了代码，但开发者需要回到本地 IDE 才能查看文件、运行终端命令、提交 Git
2. **工具分散** — 文件浏览、终端操作、版本控制分布在不同工具中，无法在统一界面内完成闭环工作流
3. **能力封闭** — 平台功能固定，开发者无法通过插件或扩展来定制自己的工作流

Hapi Power 致力于解决这些问题：在 AI 代理远程管理能力之上，构建完整的开发者工具链，让"浏览器即开发环境"成为现实。

### 1.3 核心价值

| 价值维度 | 交付能力 | 用户收益 |
|---------|---------|---------|
| **一站式工作台** | 文件管理 + 代码编辑 + 终端 + Git 管理，单一 Web 界面覆盖 | 消除工具切换，工作流闭环 |
| **AI 代理深度集成** | 代理操作与开发工具双向联动，代理修改实时反映在文件树和 Git 状态中 | 所见即所得的 AI 协作体验 |
| **可扩展生态** | 插件系统 + Skill 管理 + Claude Plugin 三层扩展架构 | 开发者可按需定制工作流 |
| **随时随地访问** | 响应式设计 + 实时通信，桌面和移动设备均可使用 | 不受设备和地点限制 |

### 1.4 核心原则

- **实用至上**：每个功能都以解决真实开发场景为导向，拒绝花架子
- **渐进增强**：模块化架构，各功能独立可用，组合后体验倍增
- **开放可扩展**：插件系统作为一等公民，平台能力通过扩展增长
- **开发者体验**：Dark Mode OLED 主题、键盘优先、终端优先，为开发者量身打造

### 1.5 致谢

本项目在开发过程中参考了以下开源项目的优秀设计：

- [hapi](https://github.com/twsxtd/hapi)（AGPL-3.0）— AI 编码代理远程管理平台，其 CLI-Hub-Web 三层架构和实时通信模式为本项目的架构设计提供了重要参考
- [dinotty](https://github.com/nicepkg/dinotty) — 移动端优先的 Web 终端服务器，其 PTY 管理和动态插件系统设计为本项目带来了重要启发

## 2. 设计参考

> 本节记录设计过程中的技术调研成果，为架构决策提供依据。

### 2.1 技术调研概况

设计前期对现代 AI 编码代理平台和 Web 终端服务器的架构模式进行了深入分析，覆盖多种编程语言和架构分层，形成完整的知识图谱。

| 调研维度 | 覆盖范围 |
|---------|---------|
| 语言生态 | TypeScript, Rust, Vue, CSS, YAML, Markdown, JavaScript, Shell, HTML, TOML, Dockerfile |
| 框架识别 | React, Vue, Vite, Axum, Tauri, Tokio, Fastify, Hono, Socket.IO, TailwindCSS, TanStack Query |
| 架构模式 | CLI-Server-Web 三层架构、实时通信（Socket.IO/SSE）、服务端虚拟终端、动态插件加载 |

### 2.2 关键架构洞察

通过调研提炼出以下架构模式，直接影响 Hapi Power 的技术选型：

**AI 代理管理平台的典型架构（Bun + TypeScript monorepo）：**

```
cli/      → AI 代理包装器 + 运行守护进程
hub/      → HTTP 服务 + 实时通信 + 数据库
web/      → React SPA + TanStack Router/Query + Tailwind CSS
shared/   → 跨包共享类型、schemas、工具函数
```

核心模式：
- **SyncEngine** — 会话/消息管理引擎
- **双通道通信** — CLI ↔ Hub（Socket.IO）↔ Web（SSE + REST API）
- **RPCGateway** — Web→CLI 远程过程调用路由

**Web 终端服务器的典型架构：**

```
server/       → 后端（PTY 管理 + 会话广播）
frontend/     → SPA（xterm.js + Monaco Editor）
plugins/      → 插件系统
plugin-api/   → 插件 API 类型定义
```

核心模式：
- **服务端 VirtualScreen** — @xterm/headless 虚拟终端，多客户端广播
- **动态插件加载** — Blob URL import() + PluginContext API
- **文件工作区** — 树形浏览 + Monaco 编辑 + 语法检查

### 2.3 可复用的成熟组件

调研中识别出可直接复用的生产级组件：

| 能力 | 组件方向 | 技术基础 |
|------|---------|---------|
| Diff 查看 | Git diff 渲染 + Monaco DiffEditor | react-diff-view + Shiki |
| 代码高亮 | 语法着色引擎 | Shiki（多语言支持） |
| 终端模拟 | xterm.js 终端渲染 + PTY 管理 | node-pty + @xterm/headless |
| 文件浏览 | 树形文件浏览器 | react-complex-tree（懒加载 + 虚拟化） |
| 实时更新 | SSE 推送 + WebSocket 双向通信 | Socket.IO + SSE |
| 插件系统 | 动态加载 + 沙箱隔离 | Blob URL import() + ErrorBoundary |

## 4. 设计系统

### 4.1 视觉风格

基于 Cursor + Linear 融合风格，核心理念：**更深、更密、更快**。

| 维度 | 选择 | 说明 |
|------|------|------|
| **风格** | Dark Mode (OLED) | 近纯黑背景 #0A0A0B，开发者工具首选 |
| **模式** | 纯暗色主题 | 与平台暗色主题保持一致 |
| **参考** | Cursor + Linear 融合 | Cursor 的密度与操作效率 + Linear 的层次与精致感 |
| **图案** | Real-Time / Operations | 数据密集但可扫描，状态色彩明确 |
| **圆角** | 6px 统一 | 按钮、卡片、面板、输入框统一 6px，小型元素 4px |
| **性能** | 优秀 | 高对比度深色主题渲染效率高 |
| **可访问性** | WCAG AAA | 高对比度，符合最高标准 |

#### 风格特征

- **背景层次**：3 级深度 — Canvas (#0A0A0B) → Surface (#141416) → Elevated (#1C1C1F)
- **信息密度**：紧凑布局，行高 1.5，间距 4/8/12/16px 阶梯，参考 Cursor 的面板布局
- **交互反馈**：hover 背景色微变、active 缩放 0.98、focus 绿色焦点环，参考 Linear 的过渡流畅感
- **终端风格**：参考 Warp 的"命令块"设计，每个命令及其输出构成独立视觉单元
- **动画体系**：50ms（即时响应）→ 150ms（微交互）→ 250ms（面板过渡），系统级动画变量

### 4.2 色彩体系

#### 背景层次（3 级深度）

| Surface | `#141416` | `--color-surface` | 卡片/面板/侧边栏背景 |
| Elevated | `#1C1C1F` | `--color-elevated` | 悬浮面板/弹窗/下拉背景 |

#### 前景与边框

| 角色 | 色值 | CSS 变量 | 用途 |
|------|------|----------|------|
| Foreground | `#E6EDF3` | `--color-foreground` | 主要文字 |
| Muted Text | `#8B949E` | `--color-muted-text` | 弱化文字（对比度 5.5:1） |
| Faint | `#484F58` | `--color-faint` | 极弱元素（分割线、占位符） |
| Border | `#30363D` | `--color-border` | 边框 |
| Border Focus | `#22C55E` | `--color-border-focus` | 聚焦态边框 |

| 语义 | 色值 | CSS 变量 | 用途 | 场景 |
|------|------|----------|------|------|

| 状态 | 色值 | 说明 |

### 4.4 关键效果

#### 动画变量体系

```css
:root {
  /* 时长阶梯 */
  --duration-instant: 50ms;     /* 按钮按下、开关切换 */
  --duration-fast: 150ms;       /* hover、微交互 */
  --duration-normal: 250ms;     /* 面板展开、弹窗出现 */
  --duration-slow: 400ms;       /* 页面过渡、复杂动画 */

  /* 缓动函数 */
  --ease-default: cubic-bezier(0.25, 0.1, 0.25, 1);
  --ease-in: cubic-bezier(0.4, 0, 1, 1);
  --ease-out: cubic-bezier(0, 0, 0.2, 1);
  --ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1);

  /* 圆角 */
  --radius-sm: 4px;
  --radius-md: 6px;
  --radius-lg: 10px;
  --radius-full: 9999px;

  /* 阴影（层级感） */
  --shadow-sm: 0 1px 2px rgba(0,0,0,0.3);
  --shadow-md: 0 4px 12px rgba(0,0,0,0.4);
  --shadow-lg: 0 8px 24px rgba(0,0,0,0.5);
}
```

#### 交互反馈规范

| 交互类型 | 效果 | 时长 | 缓动 |
|---------|------|------|------|
| Hover | 背景 `rgba(255,255,255,0.04)` | 150ms | ease-out |
| Active | 缩放 0.98 + 背景加深 | 50ms | instant |
| Focus | 2px 焦点环 `--color-accent` | 150ms | ease-out |
| 面板展开 | translateX + opacity | 250ms | ease-spring |
| 弹窗出现 | scale(0.95→1) + opacity | 250ms | ease-out |
| Toast | translateY + opacity | 250ms | ease-spring |

#### 可访问性要求

- 高可读性：所有文字对比度 >= 4.5:1（AA），标题 >= 3:1
- `prefers-reduced-motion` 支持：关闭所有动画，仅保留 opacity 过渡
- 键盘焦点环始终可见，2px 实线 `--color-accent`
- 微光效果仅用于状态指示器，reduced-motion 下关闭

### 4.5 响应式断点

| 名称 | 断点 | 布局策略 |
|------|------|---------|
| Mobile | < 640px | 单栏堆叠，底部导航 |
| Tablet | 640px - 1023px | 双栏折叠，侧边导航 |
| Desktop | 1024px - 1439px | 多栏展开，侧边导航 |
| Wide | >= 1440px | 最大宽度容器，侧边导航 |

### 4.6 设计禁忌

- 不使用浅色主题作为默认
- 不使用 emoji 作为图标（使用 Lucide SVG 图标）
- 不使用缓慢的加载动画（> 400ms 需评审）
- 不混用不同的视觉风格
- 不使用对比度不足的色彩组合（需 WebAIM 验证）
- 不使用超过 3 级的背景层次（Canvas → Surface → Elevated）
- 不使用非变量字重的静态 Inter 字体（必须用 Inter Variable）
- 不使用硬编码的动画时长（必须引用 CSS 变量）
- 不使用不一致的圆角值（必须引用 --radius 变量）

### 4.7 已有功能风格适配

| 字体迁移 | 全局 | Inter Variable 变量字体，支持全字重轴 |

| 语义色统一 | 全局 | 5 色语义系统（success/warning/danger/info/accent） |
| 动画变量化 | 全局 | 系统级动画变量（duration + easing + radius） |
| 终端样式增强 | Terminal | Warp 风格命令块视觉效果 |

### 7.1 国际化（i18n）架构

#### 方案选型

自建轻量 i18n 方案，不引入第三方框架。项目已有 I18nProvider + useTranslation hook 基础设施，在此之上增强。

#### 核心实现

```typescript
// 1. Provider + Hook（已有基础）
interface I18nConfig {
  locale: string           // 当前语言，默认 'zh-CN'
  fallbackLocale: string   // 回退语言 'en'
  messages: Record<string, Record<string, string>>  // 命名空间化消息
}

// 2. 翻译文件组织
web/src/i18n/
├── locales/
│   ├── zh-CN/           # 默认语言
│   │   ├── common.json  # 通用文案（按钮、状态、错误）
│   │   ├── git.json     # Git 模块
│   │   ├── files.json   # 文件管理模块
│   │   ├── terminal.json # 终端模块
│   │   ├── extensions.json # 扩展模块
│   │   └── settings.json  # 设置模块
│   └── en/
│       └── ...          # 英文翻译（同结构）
├── index.ts             # I18nProvider + useTranslation
└── types.ts             # 类型定义
```

#### 增强项

| 增强项 | 方案 | 说明 |
|-------|------|------|
| 命名空间拆分 | 按模块拆分 JSON | 按需加载，避免单文件过大 |
| 复数支持 | `Intl.PluralRules` | 支持 `{count, plural, one{...} other{...}}` 语法 |
| 后端通知模板 | 错误码 + i18n key | 后端返回 code，前端根据 locale 渲染消息 |
| 语言切换 | 热切换 | 不刷新页面，Context 驱动重渲染 |
| 默认语言 | zh-CN | 中文优先 |
| 回退语言 | en | 缺失翻译回退英文 |

#### 翻译 Key 规范

```
{module}.{section}.{element}.{state}
例：
  git.status.staged          → "已暂存"
  git.status.unstaged        → "未暂存"
  files.action.create.success → "文件创建成功"
  terminal.session.reconnect  → "正在重新连接终端..."
  common.button.save          → "保存"
  common.error.network        → "网络连接失败"
```

#### 支持语言

| 语言 | 代码 | 状态 | 优先级 |
|------|------|------|-------|
| 简体中文 | zh-CN | 默认语言，完整覆盖 | P0 |
| English | en | 完整覆盖 | P0 |

## 9. 架构决策记录（ADR）

> 基于两轮专业评审（UI/UX + 前端架构 + 系统架构 + 安全）的 7 项 CRITICAL / 13 项 HIGH 问题，以下为最终决策。
> 完整评审报告见 [v0.1-review-report.md](./v0.1-review-report.md)。

### 9.1 ADR-001：Socket.IO 统一传输层

**问题**：Module B 使用 Socket.IO `/pty` 命名空间，Module F.3 定义了独立的 raw WebSocket 二进制帧协议，两者冲突。

**决策**：Socket.IO 作为唯一实时传输层。二进制帧（终端流、文件传输）通过 Socket.IO 的 binary event 支持（`socket.emit('binary', Buffer)`）在同一连接上传输。

**理由**：Socket.IO 原生支持 binary frame（自动降级为 ArrayBuffer），避免双连接管理、认证分裂和连接状态不一致。二进制帧的 opcode + slot 首字节编码格式不变，只是载体从 raw WebSocket 换成 Socket.IO binary event。

**影响**：Module B 的 `/pty` 命名空间扩展为支持二进制事件；Module F.3 的 WebSocket 连接改为 Socket.IO binary emit。

### 9.2 ADR-002：Session 中心隔离模型

**问题**：Module A 使用独立 `/api/git/*` 路由 + userId 隔离，其他模块使用 `/sessions/:id/` + sessionId 隔离，数据隔离模型不一致。

**决策**：所有模块统一为 session 中心模型。Git 模块在 `git_repos` 表增加 `session_id` 字段关联 session。API 路由统一前缀 `/api/sessions/:id/...`，内部验证 sessionId → userId 归属。

**理由**：Hapi Power 的核心操作单位是 session（一次 AI 代理对话），而非 userId。多 CLI 实例场景下，同一用户的不同 session 操作不同工作目录，按 session 隔离更精确。

**影响**：Module A 路由从 `/api/git/*` 调整为 `/api/sessions/:id/git/*`；所有 API 统一添加 session 归属校验。

### 9.3 ADR-003：Hub 事件总线

**问题**：模块间缺乏事件通知机制，Git 文件变更无法通知文件管理器，代理状态变化无法触发消息出队。

**决策**：在 Hub 核心引擎中增加 `EventBus`（基于 Node.js EventEmitter），定义标准化事件协议。

```typescript
// 事件协议
interface HubEvent<T = unknown> {
  type: string          // 'file:changed' | 'git:committed' | 'agent:status' | ...
  sessionId: string
  timestamp: number
  payload: T
}

// 关键事件类型
type HubEventType =
  | 'file:changed'          // 文件内容变更（C 发出，E/F 订阅）
  | 'file:tree:updated'     // 文件树结构变更（C 发出）
  | 'git:status:changed'    // Git 状态变更（A 发出，C/E 订阅）
  | 'agent:status'          // 代理状态变更（F 发出，F.1 订阅）
  | 'context:compressed'    // 上下文压缩（G 发出，前端订阅）
  | 'pty:created'           // PTY 创建（B 发出）
  | 'plugin:loaded'         // 插件加载（D 发出）
```

**理由**：EventEmitter 零依赖、同步调用、类型安全（TypeScript 泛型约束）。Hub 是单进程架构（SQLite），不需要跨进程消息队列。

**影响**：各模块通过 `eventBus.emit()` 发布事件，通过 `eventBus.on()` 订阅。WebSocket/SSE 负责将事件推送到前端。

### 9.4 ADR-004：Git 操作统一管理

**问题**：Module E.3 的撤销变更设计了独立的 auto-commit 策略，与 Module A 的 Git 管理冲突。

**决策**：所有 Git 操作（包括 auto-commit）由 Module A 统一管理。Module E.3 通过 Module A 暴露的 `autoCommit(sessionId, description)` API 间接操作。

```typescript
// Module A 暴露的内部 API
interface GitInternalAPI {
  autoCommit(sessionId: string, message: string): Promise<string | null>
  // 返回 commit hash，无 Git 仓库时返回 null
  getFileAtCommit(sessionId: string, commitHash: string, filePath: string): Promise<string | null>
  resetToCommit(sessionId: string, commitHash: string): Promise<void>
}
```

**理由**：单一 Git 操作入口避免冲突、保证状态一致。Module A 负责判断是否有 Git 仓库、是否需要 stash 保护、commit hash 记录。

**影响**：Module E.3 不再直接调用 `git` 命令，改为调用 Module A 的内部 API。无 Git 仓库时自动降级为文件快照方案。

### 9.5 ADR-005：代码编辑器归入文件管理

**问题**：架构图中包含 CodeEdit/Preview，但没有对应模块文档覆盖文件内容读写 API 和 Monaco Editor 集成。

**决策**：代码编辑器作为 Module C（文件管理）的子节，不创建独立模块文档。文件读写 API 归入文件管理路由。

**理由**：代码编辑器本质是文件内容的查看/编辑能力，与文件树浏览属于同一用户流程（选择文件 → 查看 → 编辑 → 保存），分离反而增加导航成本。
