# Hapi Power v0.18.0 Five Modules

v0.18.0 turns Hapi Power into a five-step AI coding workbench: connect models, steer running agents, observe context risk, trace code changes, and preserve session knowledge.

| Module | Entry | Primary output |
|--------|-------|----------------|
| Model Nexus | Settings -> Model Nexus | Provider health, model cache, and default agent routes |
| Guide Beam | Session composer while the agent is thinking | Immediate guidance that preserves the normal queue |
| Context Pulse | Session status bar | `Context: n%` usage signal with source diagnostics |
| Git Atlas | Session -> Git | Branch state, change map, diff preview, commit basket, and sync controls |
| Session Loom | Session -> Loom | Outline, Markdown exports, background design synthesis, and export assets |

## Model Nexus

Model Nexus is the provider control plane. It stores namespace-scoped providers, checks health, discovers model lists, and assigns provider/model routes to Claude, Codex, Gemini, OpenCode, Cursor, and Kimi.

Key behavior:

- Supports `auto`, `openai`, `anthropic`, and `gemini` provider protocols.
- Validates provider URLs before saving and blocks unsafe schemes, userinfo, private or metadata IPs, unsafe redirects, and non-standard ports unless explicitly allowed by administrator policy.
- Encrypts API keys with AES-256-GCM; decrypted keys can only be shown through a short explicit reveal-token flow.
- Caches model discovery results and exposes a route matrix so each agent can have a default provider and model.
- Keeps safe diagnostics only, so failed checks do not leak API keys or full upstream responses.

Implementation references:

- Hub routes: `hub/src/web/routes/providers.ts`
- Discovery and SSRF protection: `hub/src/services/modelDiscovery.ts`, `hub/src/services/providerSecurity.ts`
- Web UI: `web/src/components/ProviderSettings.tsx`
- Shared types: `shared/src/providers.ts`

## Guide Beam

Guide Beam gives the user a steering channel while an agent is already working. Normal follow-up messages stay queued, while Guide now attempts an isolated immediate delivery.

Key behavior:

- Web sends `deliveryMode: "guide"` only for guide messages.
- Hub emits a `guide-message` only when the session is thinking, the connected CLI socket has the `guideInterrupt` capability, the capability preserves the queue, delivery is isolated, and no permission request is pending.
- If those conditions are not met, the message falls back to the ordinary queue with a visible fallback reason.
- Guide messages cannot be scheduled and cannot include attachments; this prevents ambiguous delivery and missing uploaded files.
- CLI queues consume guide messages alone before normal queued batches, so a correction does not merge into unrelated text.

Implementation references:

- Hub message routing: `hub/src/sync/messageService.ts`
- Session capability tracking: `hub/src/sync/syncEngine.ts`
- CLI message queue: `cli/src/utils/MessageQueue2.ts`
- Web composer and queue UI: `web/src/components/AssistantChat/ComposerButtons.tsx`, `web/src/components/AssistantChat/QueuedMessagesBar.tsx`

## Context Pulse

Context Pulse replaces vague token text with a compact reliability signal such as `Context: 40%`. It tells the user how much of the model context is currently consumed and why the number may be unavailable.

Key behavior:

- Reads normalized assistant usage from Claude and Codex streams, including `context_tokens`, prompt/input tokens, cached tokens, and model context windows.
- Uses the provider-reported `context_window` when available; otherwise it falls back to known model budgets.
- Shows threshold tones: below 60% is healthy, 60-80% is warning, above 80% is danger.
- The details popover explains used tokens, max tokens, source (`reported`, `estimated`, or `unknown`), model, cache reads, and unavailable reasons.
- The popover closes on outside click, Escape, scroll, resize, and mobile viewport movement, so it does not cover the composer permanently.

Implementation references:

- Usage normalization: `web/src/chat/normalizeAgent.ts`
- Status UI: `web/src/components/AssistantChat/StatusBar.tsx`
- Claude context propagation: `cli/src/claude/utils/sdkToLogConverter.ts`
- Codex token usage conversion: `cli/src/codex/utils/appServerEventConverter.ts`

## Git Atlas

Git Atlas is the session Git decision page. It turns raw Git commands into a scan-first workspace for reviewing and shipping agent changes.

Key behavior:

- Builds a `git-dashboard` response with repository state, current branch, upstream, ahead/behind counts, conflicts, remotes, recent commits, and recommendations.
- Groups changes into a Change Map and provides per-file diff preview with binary and large-diff fallbacks.
- Commit Basket lets users choose exact paths for focused commits.
- Sync Center supports fetch, pull, and push; force push requires typing the current branch name.
- Remote and branch management remain available as secondary controls, while dangerous operations keep server-side confirmation and sanitized output.

Implementation references:

- Hub Git routes: `hub/src/web/routes/git.ts`
- Web route: `web/src/routes/sessions/git.tsx`
- Git UI helpers: `web/src/lib/git-atlas.ts`
- Shared API types: `shared/src/apiTypes.ts`

## Session Loom

Session Loom turns the old outline panel into a session asset workbench. It has four tabs: Outline, Export, Synthesis, and Assets.

Key behavior:

- Outline is server-generated from the full session history and supports filtering by all, user, and assistant entries.
- Export is deterministic: it builds Markdown from stored messages, templates, and filters. It does not call a model.
- Export timestamps are rendered in Beijing Time, so Markdown records match the user's review timezone.
- Filters default to secret redaction on, system events off, and tool details off. Templates include raw transcript, design handoff, PRD notes, decision log, retrospective, drift check, and lesson card.
- Synthesis is different from export: it calls the current session agent's configured provider API model in the background, analyzes the full conversation, and creates a reusable Design Plan Markdown asset without inserting a message into the active chat.
- External model selection is intentionally rejected for synthesis; the session's current agent route is the source of truth.
- Assets can be downloaded, copied, shared where supported, refreshed, and deleted. The in-memory asset store keeps the newest assets per session and expires them after the configured TTL.

Implementation references:

- Hub routes and Markdown builder: `hub/src/web/routes/sessionLoom.ts`
- Background model synthesis: `hub/src/services/sessionLoomSynthesis.ts`
- Web panel and standalone page: `web/src/components/AssistantChat/SessionLoomPanel.tsx`, `web/src/routes/sessions/loom.tsx`
- Shared request and response types: `shared/src/apiTypes.ts`

## Verification

The v0.18.0 release path validates these modules with focused unit tests plus full package checks:

- Provider and discovery tests cover health checks, SSRF blocking, key reveal, assignment, and model cache behavior.
- Guide Beam tests cover capability handshakes, fallback reasons, queued-message ordering, retries, and CLI queue isolation.
- Context Pulse tests cover usage normalization, thresholds, unavailable states, and status-bar rendering.
- Git Atlas tests cover dashboard parsing, selected-path commits, sync requests, clone safety, and i18n keys.
- Session Loom tests cover outline generation, export templates, redaction, Beijing Time formatting, asset lifecycle, and background synthesis.

Run the main gates before release:

```bash
bun run typecheck
bun run test
bun run build
bun run check:git-standards
bun run check:sensitive-info
git diff --check
```
