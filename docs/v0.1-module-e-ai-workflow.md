> Skill 绑定：architect（技术）+ frontend-design（前端）
> 调研参考：Aider、OpenHands、Langfuse、LangGraph、Cursor、Claude Code
> 架构决策：ADR-003（事件总线）、ADR-004（Git 操作统一管理）

## E.1 变更审查流程

### E.1.1 问题场景

AI 代理一次对话可能修改多个文件，开发者需要：
- 看到这次对话导致了哪些文件变更
- 逐文件审查 diff，理解修改原因
- 逐文件 approve/reject，而非全部接受或拒绝
- 变更与对话上下文关联 — 点击文件能定位到对应的代理操作

### E.1.2 核心设计决策

1. **基于 ToolCall 的变更追踪** — 代理每次文件操作（Write/Edit 工具调用）都携带 `file_path`、`old_string`、`new_string`，天然支持 diff 渲染
2. **按对话分组** — 每次对话（turn）产生的文件变更为一组，`conversationId` 关联
3. **复用 DiffView 组件** — 已有 DiffView 支持预览/内联模式，直接扩展
4. **三态审查模型** — 参考 OpenHands 的 `AWAITING_CONFIRMATION / CONFIRMED / REJECTED` 状态机

### E.1.3 数据模型

```typescript
interface FileChange {
  id: string
  sessionId: string
  messageId: string              // 关联到对话消息
  filePath: string
  changeType: 'created' | 'modified' | 'deleted'
  beforeContent: string | null   // 修改前内容（null 为新建）
  afterContent: string | null    // 修改后内容（null 为删除）
  reviewStatus: 'pending' | 'approved' | 'rejected'
  reviewedAt?: number
  timestamp: number
}

interface ChangeGroup {
  sessionId: string
  conversationTurnId: string     // 一轮对话产生的所有变更
  changes: FileChange[]
  summary: string                // "修改了 3 个文件，新建 1 个文件"
  agentDescription?: string      // 代理自述的变更意图
  createdAt: number
}
```

### E.1.4 API 路由

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/sessions/:id/changes` | 获取未审查的变更列表 |
| GET | `/sessions/:id/changes/:changeId/diff` | 获取单个文件变更的 diff |
| POST | `/sessions/:id/changes/:changeId/review` | 审查单个变更（approve/reject） |
| POST | `/sessions/:id/changes/bulk-review` | 批量审查 |
| GET | `/sessions/:id/changes/history` | 变更审查历史 |

### E.1.5 前端组件树

```
ChangeReviewPanel                       (变更审查面板入口)
├── ChangeGroupList                     (按对话分组的变更列表)
│   ├── ChangeGroupCard                 (一组变更摘要)
│   │   ├── ChangeSummary               ("修改了 3 个文件")
│   │   ├── AgentDescription             (代理自述意图，可折叠)
│   │   └── FileChangeList              (文件变更列表)
│   │       ├── FileChangeItem          (单文件：状态徽章 + 路径 + 操作数)
│   │       │   ├── ReviewStatusBadge   (pending/approved/rejected)
│   │       │   ├── DiffStats           (+42/-18)
│   │       │   └── QuickActions        (approve/reject 按钮)
│   │       └── DiffPreview             (展开后的 diff 视图)
│   │           └── DiffView            (复用现有组件，内联模式)
│   └── BulkActions                     (全部批准 / 全部拒绝)
└── ChangeReviewToolbar                 (过滤：按状态/文件类型)
```

### E.1.6 交互设计

1. **入口** — 对话面板顶部，当有未审查变更时显示徽章计数："3 个文件待审查"
2. **快速审查** — 文件列表中直接点击 approve/reject，无需展开 diff
3. **详细审查** — 点击文件展开 diff，查看行级变更
4. **上下文跳转** — 点击文件路径，Chat 面板自动滚动到对应的 ToolCall 消息
5. **批量操作** — "全部批准"需二次确认，"全部拒绝"需确认 + 可选理由

### E.1.7 安全考虑

- reject 操作仅撤销审查标记，不自动还原文件（还原由"撤销变更"模块处理）
- approve 仅标记审查通过，不自动 commit
- 变更记录不可删除，保证审计追踪完整

## E.2 代理操作时间线 + 会话提炼总结

### E.2.1 问题场景

开发者离开 2 小时回来，需要快速了解：
- 代理做了什么（文件读写、命令执行、自动批准）
- 代理遇到了什么问题
- 当前状态是什么（进行中/已完成/等待审批）
- 关键决策和上下文是什么

### E.2.2 核心设计决策

1. **复用消息流基础设施** — 时间线数据来源是已有的 ToolCall 消息，不需要额外的追踪系统
2. **结构化提取** — 从消息流中提取操作记录，按类型分类（文件操作、命令执行、权限审批）
3. **增量摘要** — 参考 LangGraph 的 `summarize_conversation` 模式，增量维护摘要而非全量重新生成
4. **检查点机制** — 参考 Cursor 的 6 步工作流，在代理操作的关键节点自动创建检查点

### E.2.3 数据模型

```typescript
// 操作时间线条目（从消息流中提取）
interface TimelineEntry {
  id: string
  sessionId: string
  messageId: string               // 关联到原始消息
  timestamp: number
  type: 'file_read' | 'file_write' | 'file_delete'
     | 'command_exec' | 'command_result'
     | 'permission_request' | 'permission_granted' | 'permission_denied'
     | 'llm_call' | 'llm_response'
     | 'checkpoint'
  target: string                  // 文件路径 / 命令字符串 / 工具名
  result: 'success' | 'failure' | 'pending'
  durationMs?: number
  summary: string                 // "读取 src/auth.ts（42 行）"
  metadata?: Record<string, unknown>
}

// 会话摘要
interface SessionSummary {
  id: string
  sessionId: string
  generatedAt: number
  trigger: 'auto_threshold' | 'session_end' | 'manual'
  content: {
    overview: string              // "本次会话完成了用户认证模块的重构"
    decisions: string[]           // ["选择 JWT 而非 session cookie"]
    architectureChanges: string[] // ["新增 UserRepository 层"]
    filesModified: Array<{
      path: string
      action: 'created' | 'modified' | 'deleted'
      linesAdded: number
      linesRemoved: number
    }>
    commandsExecuted: Array<{
      command: string
      result: 'success' | 'failure'
    }>
    pendingItems: string[]        // ["错误处理尚未完成"]
    tokenUsage?: {
      input: number
      output: number
      estimatedCost: number
    }
  }
  previousSummaryId?: string     // 增量摘要链
}
```

### E.2.4 API 路由

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/sessions/:id/timeline` | 获取操作时间线（支持 type 过滤） |
| GET | `/sessions/:id/summary` | 获取当前会话摘要 |
| POST | `/sessions/:id/summary/generate` | 手动触发生成摘要 |
| GET | `/sessions/:id/summary/history` | 摘要历史（增量链） |
| POST | `/sessions/:id/summary/save-to-memory` | 保存摘要到项目记忆 |

### E.2.5 时间线前端组件树

```

├── TimelineHeader                      (会话概览 + 摘要)
│   ├── SessionStatusIndicator          (进行中/已完成/等待审批)
│   ├── TokenUsageBar                   (token 使用量可视化)
│   └── QuickSummary                    (一句话摘要，可展开)
├── TimelineFilter                      (过滤工具栏)
│   ├── TypeFilter                      (文件操作/命令/权限/LLM 调用)
│   └── ResultFilter                    (成功/失败/待处理)
└── TimelineList                        (时间线条目列表)
    └── TimelineItem                    (单条操作)
        ├── TimelineIcon                (操作类型图标 + 颜色)
        ├── TimelineContent             (操作摘要 + 时间戳)
        │   └── InlineDiff              (文件操作：行级变更预览)
        ├── TimelineMeta                (持续时间、token 消耗)

```

### E.2.6 摘要触发策略

| 触发条件 | 方式 | 说明 |
|---------|------|------|
| 消息数 > 50 条 | 自动 | 基于阈值自动触发增量摘要 |
| 会话结束 | 自动 | 代理主动结束时生成完整摘要 |
| 手动触发 | 用户操作 | 类似 `/summary` 命令 |
| 开发者离开 | 推测 | 页面失焦 > 30 分钟后回来，显示摘要 |

### E.2.7 摘要生成流程

```
1. 收集本轮消息（自上次摘要以来的新消息）
2. 调用 LLM 生成结构化摘要（JSON schema 约束输出格式）
3. 与上轮摘要合并（增量模式）
4. 存储到 SessionSummary 表
5. 前端展示摘要卡片
```

### E.2.8 检查点机制

在以下时机自动创建检查点：
- 代理完成一个完整操作序列（如一组文件修改）
- 代理执行命令前后
- 权限审批决策前后

检查点数据轻量化：仅记录文件状态快照（哈希值），不存储完整内容。完整内容回溯通过 Git 或文件快照系统获取。

## E.3 撤销变更

### E.3.1 问题场景

- 代理搞砸了，想回到"代理开始之前"的状态
- 代理中间某步操作有问题，想撤销那一步
- 只想撤销某个文件的修改，保留其他

### E.3.2 核心设计决策

1. **双层回滚机制** — Git 优先（通过 Module A 内部 API，ADR-004），内存快照兜底
2. **三种粒度** — 会话级（撤销整个对话的变更）、步骤级（撤销某次操作）、文件级（撤销单个文件）
3. **安全前置** — 回滚前自动创建备份，支持"撤销撤销"
4. **Git 操作由 Module A 统一管理** — 不直接调用 git 命令，通过 GitInternalAPI 间接操作（ADR-004）

### E.3.3 数据模型

```typescript
// 会话级变更追踪
interface SessionChangeTracker {
  sessionId: string
  gitCommitHashes: string[]       // AI 操作产生的 git commit（Aider 模式）
  fileSnapshots: Map<string, {    // 操作前文件快照（内存模式）
    before: string
    after: string
    timestamp: number
    messageId: string
  }>
  checkpoints: SessionCheckpoint[]
}

interface SessionCheckpoint {
  id: string
  sessionId: string
  label: string                   // "代理开始前" / "文件修改前" / "命令执行前"
  timestamp: number
  type: 'auto' | 'manual'
  fileHashes: Record<string, string>  // filePath → content hash
  gitHead?: string                // 当前 HEAD commit hash
  messageIndex: number            // 对话消息位置
}

// 撤销操作记录
interface UndoRecord {
  id: string
  sessionId: string
  checkpointId: string            // 回滚到哪个检查点
  scope: 'session' | 'step' | 'file'
  targetFiles: string[]           // 受影响的文件列表
  undoMethod: 'git_reset' | 'git_checkout' | 'file_restore'
  performedAt: number
  backupCheckpointId: string      // 回滚前的备份检查点
}
```

### E.3.4 回滚策略

```
优先级判断流程：
1. 工作目录是 Git 仓库？
   → 是：使用 Git 回滚（可靠、可审计）
   → 否：使用文件快照回滚（基于内存检查点）

2. Git 回滚策略：
   - 文件级：git checkout <checkpoint-hash> -- <file>
   - 步骤级：git checkout <checkpoint-hash> -- <affected-files>
   - 会话级：git reset --soft <session-start-commit>

3. 文件快照回滚策略：
   - 直接恢复快照内容到文件
   - 创建回滚前的备份快照
```

### E.3.5 API 路由

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/sessions/:id/checkpoints` | 列出可用检查点 |
| POST | `/sessions/:id/checkpoints` | 手动创建检查点 |
| GET | `/sessions/:id/checkpoints/:id/preview` | 预览回滚影响（将变更的文件列表） |
| POST | `/sessions/:id/undo` | 执行撤销（指定检查点 + 范围） |
| GET | `/sessions/:id/undo/history` | 撤销历史 |
| POST | `/sessions/:id/undo/:undoId/redo` | 重做（撤销撤销） |

### E.3.6 前端交互

1. **一键撤销** — 对话面板顶部 "撤销本次对话变更" 按钮
2. **时间线回滚** — 时间线面板中，每个检查点旁有 "回滚到此点" 按钮
3. **单文件撤销** — 变更审查面板中，单个文件有 "还原此文件" 按钮
4. **预览确认** — 执行回滚前弹出预览："将还原 5 个文件，创建备份检查点"，需确认
5. **回滚后提示** — "已回滚到 [检查点名]。如需恢复，点击重做。"

### E.3.7 自动 Commit 策略（通过 Module A）

```
代理操作流程（ADR-004，不直接调用 git 命令）：
1. 代理请求修改文件
2. Module E 调用 GitInternalAPI.autoCommit(sessionId, "代理操作前快照")
   → Module A 检查工作目录 git 状态
   → 如有未提交变更 → git stash
   → Module A 返回 commit hash（无 Git 仓库时返回 null）
3. 执行文件修改
4. Module E 调用 GitInternalAPI.autoCommit(sessionId, "[hapi-power] <agent description>")
5. 记录 commit hash 到 SessionChangeTracker.gitCommitHashes

撤销流程：
1. 计算 SessionChangeTracker.gitCommitHashes.length = N
2. 调用 GitInternalAPI.resetToCommit(sessionId, startCommitHash, 'mixed')
   → Module A 执行 git reset --mixed
3. 开发者审查后决定 commit 或 checkout
```

## E.4 移动端代码文档速览

### E.4.1 问题场景

手机上最常用的操作不是写代码，而是：
- 收到推送："代理请求执行命令"，快速审批
- 快速看一眼代理改了什么
- 查看项目文档
- 查看终端输出

### E.4.2 核心设计决策

1. **专用移动路由** — 独立的 `/m/*` 路由，专为核心移动场景优化
2. **PWA 推送通知** — Web Push API 驱动审批通知
3. **聚焦审查流** — 参考 GitHub Mobile 的 PR 审查体验：文件列表 → diff → approve/reject
4. **终端只读** — 移动端终端仅显示最近输出，不支持输入（安全考虑）

### E.4.3 移动端路由

```
/m                          → 移动首页（活跃会话列表 + 待审批计数）
/m/sessions/:id             → 会话摘要（QuickSummary + 操作概览）
/m/sessions/:id/changes     → 变更审查（文件列表 + diff + approve/reject）
/m/sessions/:id/changes/:fileId → 单文件 diff 详细审查
/m/sessions/:id/terminal    → 终端输出（只读，最近 200 行）
/m/sessions/:id/timeline    → 操作时间线（精简版）
/m/approvals                → 全局审批队列（跨会话聚合）
```

### E.4.4 前端组件树

```

│   ├── SessionCardList                  (活跃会话列表)

│   └── PendingApprovalBadge            (全局待审批计数)
```

## E.5 会话分享

### E.5.1 问题场景

- 遇到 bug，想分享当前终端状态给同事求助
- 想展示代理的操作过程（教学/演示）
- 团队成员需要查看代理的对话和变更

### E.5.2 核心设计决策

1. **只读分享** — 分享链接的访问者只能查看，不能操作
2. **范围控制** — 可选择分享内容范围（对话、文件变更、终端输出）
3. **时效控制** — 分享链接可设置过期时间
4. **轻量实现** — 不需要实时协作，快照分享即可

### E.5.3 数据模型

```typescript
interface SharedSession {
  id: string
  token: string                     // 分享链接 token（UUID）
  sessionId: string
  createdBy: string                 // userId
  createdAt: number
  expiresAt: number | null          // null = 永不过期
  scope: {
    conversation: boolean           // 包含对话消息
    fileChanges: boolean            // 包含文件变更和 diff
    terminalOutput: boolean         // 包含终端输出快照
    timeline: boolean               // 包含操作时间线
  }
  accessCount: number               // 访问次数
  lastAccessedAt: number | null
}

// 分享视图（访问者看到的）
interface SharedSessionView {
  sessionName: string
  createdAt: number
  summary: SessionSummary           // 会话摘要
  messages?: Array<{                // scope.conversation = true 时
    role: string
    content: string
    timestamp: number
  }>
  changes?: Array<{                 // scope.fileChanges = true 时
    filePath: string
    changeType: string
    diff: string                    // unified diff 格式
  }>
  terminalOutput?: string           // scope.terminalOutput = true 时
  timeline?: TimelineEntry[]        // scope.timeline = true 时
}
```

### E.5.6 分享链接设计

```
格式：https://<host>/share/<uuid-token>
示例：https://example.com/share/a1b2c3d4-e5f6-7890-abcd-ef1234567890

特性：
- 无需登录即可访问
- Token 不可猜测（UUID v4）
- 支持密码保护（可选，v0.2）
- 访问次数限制（可选，v0.2）
```

### E.5.7 安全考虑

- 分享内容在创建时快照，后续会话变更不自动同步
- 脱敏选项：分享前可选择隐藏文件路径中的敏感部分
- 分享链接访问不消耗认证配额
- 速率限制：匿名访问 `/share/*` 路由 60 次/小时

## E.10 安全增强

1. **分享链接 Token**：使用 crypto.randomUUID() 生成，不可猜测
2. **分享速率限制**：匿名访问 `/share/*` 路由 60 次/小时
3. **分享内容快照**：创建时快照，后续会话变更不自动同步
4. **移动端终端只读**：PTY 输出仅显示最近 200 行，不支持输入
5. **批量审查确认**："全部批准" 需二次确认，"全部拒绝" 需确认 + 可选理由
6. **撤销安全**：回滚前自动创建备份检查点，支持重做
