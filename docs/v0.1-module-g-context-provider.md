# 模块 G：会话上下文管理详细设计

> Skill 绑定：architect（技术）+ frontend-design（前端）
> 核心痛点：上下文用量不可见、自动压缩不可知

## G.1 核心问题

用户使用 AI 代理时最常遇到的两个痛点：

1. **上下文用了多少？** — 不知道当前对话占了多少 token，不知道还剩多少空间，直到模型报错才知道溢出了
2. **有没有自动压缩？** — 代理自动压缩了上下文，但用户完全不知道，可能导致关键信息丢失

## G.2 设计目标

- 用户随时能看到上下文用量（用了多少 / 总共多少）
- 上下文压缩时，用户能明确感知（什么时候压缩的、压缩了什么）
- 用户能手动触发压缩或调整压缩策略

## G.3 数据模型

```typescript
// 上下文窗口状态
interface ContextState {
  sessionId: string
  modelId: string
  contextWindow: number              // 模型总上下文窗口（token）
  usedTokens: number                 // 已使用 token
  messageCount: number               // 消息条数
  summaryCount: number               // 已被摘要替代的消息数
  lastCompressedAt: number | null    // 上次压缩时间
  status: 'normal' | 'warning' | 'critical'
}

// 压缩事件记录
interface CompressionEvent {
  id: string
  sessionId: string
  triggeredBy: 'auto' | 'manual'
  previousTokenCount: number
  afterTokenCount: number
  compressedMessageCount: number     // 被压缩的消息数
  summaryPreview: string             // 摘要预览文本（前 200 字）
  timestamp: number
}
```

## G.4 上下文状态判定

```typescript
function getContextStatus(state: ContextState): 'normal' | 'warning' | 'critical' {
  const usage = state.usedTokens / state.contextWindow
  if (usage >= 0.9) return 'critical'       // ≥ 90%，即将溢出
  if (usage >= 0.7) return 'warning'        // ≥ 70%，注意用量
  return 'normal'
}
```

## G.5 自动压缩策略

```
触发条件：上下文使用率 ≥ 80%

压缩流程：
1. 保留最近 5 轮对话（不压缩）
2. 将更早的对话生成结构化摘要
3. 摘要替代原始消息（原始消息归档，不删除）
4. 发送压缩通知给用户

摘要内容要求：
- 保留关键决策和技术方案
- 保留用户明确提出的约束
- 保留未完成的事项
- 省略过程性对话（问候、确认、小修改）
```

## G.6 API 路由

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/sessions/:id/context` | 获取上下文状态（token 用量、消息数、压缩状态） |
| POST | `/sessions/:id/context/compress` | 手动触发压缩 |
| GET | `/sessions/:id/context/events` | 获取压缩事件历史 |

## G.7 前端组件

```
ContextIndicator                        (上下文指示器，对话面板顶部)
├── TokenBar                            (Token 用量条)
│   ├── ProgressBar                     (进度条，三段颜色：绿→黄→红)
│   ├── TokenLabel                      ("32K / 200K")
│   └── StatusIcon                      (✓ 正常 / ⚠ 注意 / ✖ 即满)
├── CompressionNotice                   (压缩通知，有压缩时显示)
│   ├── Timestamp                       ("3 分钟前自动压缩"）
│   ├── CompressedInfo                  ("压缩了 15 条消息")
│   └── SummaryToggle                   (展开/折叠摘要内容)
└── ManualCompressButton                (手动压缩按钮)
```

### 交互设计

1. **进度条颜色** — 绿色（< 70%）→ 黄色（70%-90%）→ 红色（≥ 90%）
2. **自动压缩通知** — 压缩发生时，顶部短暂显示通知条，点击可展开查看摘要内容
3. **手动压缩** — 点击压缩按钮，弹出确认"将压缩 15 条较早消息为摘要"，确认后执行
4. **用量悬浮提示** — 悬浮进度条显示详细："已用 32,000 token（16%），剩余 168,000 token"

## G.9 数据库迁移

```sql
-- 压缩事件记录表
CREATE TABLE context_compression_events (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  triggered_by TEXT NOT NULL CHECK(triggered_by IN ('auto', 'manual')),
  previous_token_count INTEGER NOT NULL,
  after_token_count INTEGER NOT NULL,
  compressed_message_count INTEGER NOT NULL,
  summary_preview TEXT,
  timestamp INTEGER NOT NULL
);

CREATE INDEX idx_compression_session ON context_compression_events(session_id, timestamp);
```

## G.10 实施

归属 Phase 1，与基础设施同期实施。核心实现：

1. 监听代理消息流，累计 token 计数（从模型返回的 usage 字段获取）
2. 达到 80% 阈值时自动压缩，发送通知
3. 前端显示用量进度条 + 压缩通知
