# 模块 D：插件系统 + Skill 管理 + Claude Plugin 管理详细设计

> Skill 绑定：architect（技术）+ frontend-design（前端）

## D.1 架构概览

三个模块共享统一的"扩展性"领域，概念独立但共享导航入口：

```
~/.claude/
├── skills/                          # Skill 管理目标
├── plugins/
│   ├── marketplaces/                # Claude Plugin 市场 git 仓库
│   ├── cache/                       # 下载的插件包
│   ├── installed_plugins.json       # 插件安装清单
│   └── local/                       # 自定义插件系统
│       ├── registry.json            # 插件注册表
│       └── {plugin-id}/            # 插件安装目录
```

### 安全模型

| 层级 | 机制 | 说明 |
|------|------|------|
| 权限声明 | plugin.json permissions | 插件必须预声明所有能力 |
| ErrorBoundary 隔离 | 每个插件面板独立 ErrorBoundary | 插件崩溃不影响主界面和其他插件 |
| 运行时权限网关 | PluginContext API 权限校验 | 每个 API 调用验证 plugin.json 声明的权限，未声明则拒绝 |
| Blob URL 隔离 | import(blobUrl) | 防止直接文件系统访问 |
| 上下文注入 | React hooks 注入 | 插件只能看到显式注入的 API |
| API 网关 | Hub 认证 | 根据 permissions 检查权限 |
| 存储隔离 | 每插件独立命名空间 | `~/.claude/plugins/local/.storage/{pluginId}.json` |
| CSP 限制 | Content-Security-Policy | 限制插件的网络请求范围 |

#### 运行时权限检查网关

```typescript
// PluginContext API 的每个方法调用前进行权限校验
class PermissionGateway {
  constructor(private manifest: PluginManifest) {}

  require(permission: string): void {
    if (!this.manifest.permissions.includes(permission)) {
      throw new PluginPermissionError(
        `Plugin ${this.manifest.id} requires '${permission}' permission`
      )
    }
  }
}

// PluginContext 实现时检查权限
const pluginContext: PluginContext = {
  terminal: {
    execute: async (command, options) => {
      gateway.require('terminal')
      return terminalService.execute(command, options)
    }
  },
  storage: {
    get: async (key) => {
      gateway.require('storage')
      return storageService.get(pluginId, key)
    }
    // ...
  }
}
```

#### ErrorBoundary 隔离

```typescript
// 每个插件面板渲染时包裹独立 ErrorBoundary
function PluginPanelRenderer({ pluginId, componentExport }: PluginPanelProps) {
  return (
    <ErrorBoundary
      fallback={<PluginCrashFallback pluginId={pluginId} />}
      onError={(error) => {
        logger.error(`Plugin ${pluginId} crashed:`, error)
        eventBus.emit('plugin:error', { pluginId, error })
      }}
    >
      <Suspense fallback={<PluginLoadingSkeleton />}>
        <PluginComponent pluginId={pluginId} componentExport={componentExport} />
      </Suspense>
    </ErrorBoundary>
  )
}
```

1. 从市场仓库复制到 `~/.claude/plugins/cache/{pluginId}/`

## D.6 关键权衡

| 决策 | 选择 | 理由 |
|------|------|------|
| 技能/插件操作 | Hub 通过 RPC 代理到 CLI | 多机架构设计，Hub 不直接访问 ~/.claude/ |
| 插件 UI 加载 | Blob URL 动态 import | 无需插件构建工具链，支持真正的动态加载 |
| 市场存储 | Git 仓库本地克隆 | 离线浏览、版本化索引、标准 git 工作流 |
| 扩展路由位置 | /settings/extensions/* | 遵循现有设置页面模式 |
| 权限模型 | 声明式 + 运行时网关 | 静态声明 + 运行时校验双重保护 |

## D.7 Skill 存储路径统一

编排 Skills（Module F.4）和普通 Skills 共享统一存储位置 `~/.claude/skills/`：

```
~/.claude/skills/
├── (用户自定义 Skills)           # Module D 管理的普通 Skills
└── orchestration/                # Module F.4 管理的编排 Skills
    ├── hapi-loop/
    ├── hapi-handoff/
    ├── hapi-advisor/
    ├── hapi-committee/
    └── hapi-epic/
```

编排 Skills 通过 Module D 的 Skill 管理接口统一管理（安装/卸载/查看），与普通 Skills 使用相同的安装和发现机制。
